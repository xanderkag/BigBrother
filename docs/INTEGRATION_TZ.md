# ТЗ: Интеграция parsedocs с внешними системами

**Дата:** 15 мая 2026
**Статус:** черновик
**Цель:** определить как parsedocs принимает документы из 1С / WMS / ERP / документооборота и как возвращает результат.

---

## Контекст

Сейчас parsedocs умеет одно — принять файл через веб-форму или `multipart/form-data POST /api/v1/jobs`. Этого достаточно для оператора с браузером, но не для интеграций с корпоративными системами. Нужно расширить набор каналов приёма и отдачи.

Аудитория интеграции:
- **1С Бухгалтерия / ERP** — присылает счета/УПД из «папки входящих»
- **WMS** — отгрузочные документы, ТТН
- **Документооборот** (СБИС, Контур.Диадок, EDI) — поток электронных копий
- **Email** — менеджер пересылает счёт от клиента
- **SFTP-папка** — старые системы кладут пачки сканов
- **Мобильное приложение** — фото со склада

Каждый канал диктует свой формат передачи. Внутри системы всё одинаково — превращается в `job` с файлом + метаданными.

## Варианты передачи документа на parsedocs

### А) Multipart upload (текущий, базовый)

```http
POST /api/v1/jobs HTTP/1.1
Authorization: Bearer <API_KEY>
Content-Type: multipart/form-data; boundary=...

------boundary
Content-Disposition: form-data; name="file"; filename="invoice.pdf"
Content-Type: application/pdf

<binary>
------boundary
Content-Disposition: form-data; name="document_hint"
invoice
------boundary
Content-Disposition: form-data; name="metadata"
{"source":"erp","external_id":"INV-2026-001"}
------boundary--
```

**Кому подходит:** браузер, любой современный HTTP-клиент (Postman, Python `requests`, axios).
**Кому не подходит:** 1С 8.3 (там multipart писать руками — больно), Java EE с XML-only.

---

### Б) Base64 в JSON-теле (для 1С и подобных)

**Что добавить:** новый endpoint `POST /api/v1/jobs/json`:

```http
POST /api/v1/jobs/json HTTP/1.1
Authorization: Bearer <API_KEY>
Content-Type: application/json

{
  "file": {
    "name": "invoice.pdf",
    "mime_type": "application/pdf",
    "content_base64": "JVBERi0xLjQKJ..."
  },
  "document_hint": "invoice",
  "webhook_url": "https://erp.taipit.ru/api/parsedocs/callback",
  "metadata": {
    "source": "erp",
    "external_id": "INV-2026-001"
  }
}
```

**Плюсы:**
- 1С пишет это за 5 строк (`HTTPСоединение` + `JSON.Сериализатор`)
- Можно дать примеры на любом языке
- Просто отлаживать (Postman работает на ура)

**Минусы:**
- +33% к размеру файла (base64-оверхед)
- Память: парсер JSON держит весь base64 в RAM до декодирования
- Не подходит для PDF >10 МБ — много мусора в логах, нагрузка на JSON-парсер

**Ограничения которые поставим:**
- Максимальный размер base64-поля: 67 МБ (= 50 МБ PDF после декодирования)
- При больших файлах сервер вернёт `413 Payload Too Large` с подсказкой использовать пресайнед URL

**Срок реализации:** 0.5 дня.

---

### В) Pull по ссылке (для случая «у меня файл уже где-то лежит»)

**Что добавить:** в `POST /api/v1/jobs/json` поле `file.url`:

```json
{
  "file": {
    "name": "invoice.pdf",
    "url": "https://internal-storage.taipit.ru/documents/2026-05/INV-001.pdf",
    "url_auth_header": "Bearer <token>"
  },
  "document_hint": "invoice",
  ...
}
```

parsedocs скачает файл сам по этой ссылке (с указанным заголовком авторизации), сохранит локально, обработает. Если скачивание упало — job уйдёт в `failed` с сообщением.

**Плюсы:**
- Можно интегрировать с любым хранилищем — корпоративный файлсервер, MinIO, Яндекс.Облако, шара
- Снимает ограничение по размеру (можем тянуть 200+ МБ)
- Метаданные приходят отдельно от тяжёлых байт

**Минусы:**
- parsedocs должна иметь сетевой доступ к источнику (firewall, mTLS, VPN)
- Источник должен быть надёжен (если ссылка протухла — потеряли документ)

**Ограничения:**
- Whitelist разрешённых доменов в `.env`: `ALLOWED_FILE_URL_HOSTS=internal-storage.taipit.ru,...`
- Тайм-аут на скачивание: 60 сек
- Максимальный размер: 200 МБ

**Срок реализации:** 1 день.

---

### Г) Presigned upload (для очень больших файлов)

Для документов >100 МБ — двухходовка:

```http
POST /api/v1/jobs/init HTTP/1.1
{
  "file_name": "big_packing_list.pdf",
  "mime_type": "application/pdf",
  "size_bytes": 153000000,
  "document_hint": "packing_list"
}
```

Возвращает:
```json
{
  "job_id": "...",
  "upload_url": "https://parsedocs.taipit.ru/api/v1/jobs/<id>/upload",
  "upload_token": "<one-time-token>",
  "expires_in": 600
}
```

Клиент кладёт сам файл `PUT` на этот URL (стрим). После загрузки job переходит в обработку.

**Кому надо:** редкие кейсы с 100+ МБ сканами (большие УПД, технические задания на сотни страниц).

**Срок реализации:** 1 день. Откладываем — пока спроса нет.

---

### Д) SFTP / папка-watch

Старые ERP и WMS любят «положить в папку — заберёшь сам».

**Что нужно:**
- SFTP-точка `sftp://parsedocs.taipit.ru/incoming/`
- Папка `/var/incoming/<org>/<project>/` на сервере, доступна по SFTP
- Sweeper-процесс каждые 30 секунд сканит папку, создаёт jobs из новых файлов
- После обработки файлы перемещаются в `/var/processed/<org>/<project>/<job_id>/`

**Манифест (опционально):**
В папку рядом с PDF кладётся `<filename>.meta.json`:
```json
{
  "document_hint": "TTN",
  "external_id": "TTN-2026-05-001",
  "metadata": {...}
}
```

**Плюсы:**
- Совместимо с самыми ретроградными системами
- Можно отлаживать через FileZilla
- Идеально для batch-выгрузок

**Минусы:**
- SSH/SFTP-сервер надо поднять, дать кому-то ключи
- Сложнее ACL — кто что может класть
- Атомарность: пока файл записывается, sweeper не должен брать (использовать `.tmp` суффикс или дождаться 5 сек после mtime)

**Срок реализации:** 2 дня.

---

### Е) Email-в-документ

Менеджер получает счёт на корпоративную почту, форвардит на `documents@parsedocs.taipit.ru` — система парсит вложения.

**Что нужно:**
- DNS MX-запись для домена parsedocs.taipit.ru
- Postfix или прямой IMAP-bot (поллит специальный ящик)
- Парсер вложений (mime-multipart, обработка PDF/JPG/EML)
- Маршрутизация по `email_from` или `email_to+org@parsedocs.taipit.ru`
- Возврат результата ответом на письмо (опционально)

**Плюсы:**
- Ноль интеграционной работы у клиента
- Знакомый интерфейс для бухгалтерии («просто переслал письмо»)
- Подходит для входящего потока от внешних контрагентов

**Минусы:**
- Спам/фишинг — нужен SPF/DKIM/DMARC, whitelist отправителей
- Сложно отслеживать ошибки (пользователь не понимает почему документ не обработался)
- Зависимость от почтового сервера (если упал — теряем поток)

**Срок реализации:** 3-4 дня (Postfix + парсер + интеграция с jobs).

---

### Ж) Webhook-приём из EDI / документооборота

Контур.Диадок, СБИС умеют слать webhook'и в нашу сторону когда приходит документ.

```http
POST /api/v1/integrations/diadok/incoming HTTP/1.1
Authorization: Bearer <integrationToken>
Content-Type: application/json

{
  "event": "document.received",
  "document": {
    "doctype": "UPD",
    "url": "https://diadoc.kontur.ru/api/box/.../documents/.../content",
    "metadata": {...}
  }
}
```

parsedocs скачивает по ссылке Diadoc (с их токеном), создаёт job, шлёт результат обратно через webhook.

**Срок реализации:** 1-2 дня на каждую систему (Диадок, СБИС, Эдо.Такском, …).

## Получение результата

### 1. Webhook (рекомендуется для всех интеграций)

После обработки парсер шлёт `POST` на `webhook_url` (указанный при создании job):

```http
POST <webhook_url> HTTP/1.1
Content-Type: application/json
X-DocService-Signature: sha256=<HMAC>
X-DocService-Job-Id: <uuid>
X-DocService-Attempt: 1

{
  "job_id": "...",
  "status": "done",
  "document_type": "invoice",
  "confidence": 0.92,
  "extracted": {
    "number": "СЧ-1787/1",
    "date": "2026-04-26",
    "seller": {"name": "...", "inn": "..."},
    "items": [...]
  },
  "validation_issues": [],
  "metadata": {"external_id": "INV-2026-001"}
}
```

Подпись HMAC-SHA256 с общим секретом. Ретраи с экспоненциальной задержкой (5 раз). Sweeper добивает 3 волны если получатель упал.

**Уже работает в текущей версии.**

### 2. Polling (для batch-выгрузок и legacy-систем)

```http
GET /api/v1/jobs/<job_id> HTTP/1.1
```

Возвращает текущее состояние. Клиент опрашивает раз в 5-30 секунд до получения `status` ∈ {done, failed, needs_review}.

**Уже работает.**

### 3. Message queue (для тех у кого есть Rabbit/Kafka)

```yaml
queue: parsedocs.results
routing_key: org.<organization_id>.project.<project_id>
exchange: parsedocs (topic)
```

Сообщение тот же payload что в webhook. Гарантия доставки at-least-once. Удобно когда у клиента несколько подсистем хотят слышать про один документ.

**Срок реализации:** 1-2 дня (выбрать брокер + worker для публикации).

### 4. Email-ответ (для входящих email)

Если документ пришёл по email — результат отвечает на то же письмо: `Re: Счёт-фактура от ...` с приложением JSON-сводки и (опционально) Excel-сводки.

## Безопасность

### Аутентификация интеграторов
- **API ключ** (уже есть, рекомендуется per-integration). В таблицу `provider_settings` добавляем тип `integration` с named-токенами.
- **mTLS** для критичных каналов (1С с банками). Сертификаты в `secrets.json`, проверка в nginx.
- **OAuth2 client credentials** — если будем интегрироваться через корпоративный SSO.

### Проверка файла
1. **Magic-bytes** (уже есть) — не верим `Content-Type` от клиента, читаем первые 16 байт. PDF, PNG, JPG, TIFF, BMP. Всё остальное — 415.
2. **Размер**: hard cap 50 МБ (наст в `MAX_UPLOAD_MB`), по протоколам отдельно.
3. **Сканер вирусов** (опционально): ClamAV в боковой контейнер, проверка перед записью на диск. Срок: 1 день.
4. **PII detection** (для Yandex Vision, уже есть `YANDEX_DISABLE_FOR_PII=true`): если в документе паспортные данные — не отправлять во внешние OCR.

### Логирование
- **Audit log** на каждый input job: кто отправил (caller-tag из API ключа), когда, какой канал, какая метадата. Уже есть.
- **PII в логах**: запросы пишутся без `file.content_base64`, только `file.name + sha256(content)`. Уже частично — нужно усилить.
- **Retention**: входящие файлы 30 дней, audit-логи 365 дней. Уже настроено.

## Метаданные и связь с системой клиента

Каждый job принимает `metadata` (произвольный JSON, до 4 КБ). Рекомендуемая структура:

```json
{
  "source": "erp" | "wms" | "email" | "edi" | "sftp",
  "external_id": "INV-2026-001",
  "external_url": "https://erp.taipit.ru/documents/12345",
  "tags": ["urgent", "Q2-audit"],
  "correlation_id": "<request-id из клиентской системы>",
  "received_at": "2026-05-15T10:30:00+03:00",
  "department": "buhuchet",
  "user": "ivanov@taipit.ru",
  "_force_provider_id": "claude" | "qwen-local" | ...
}
```

- `external_id` — для дедупликации (Idempotency-Key уже дедуплицирует на уровне HTTP-запроса, `external_id` для бизнес-логики). Уже есть.
- `correlation_id` — пробрасывается в webhook → позволяет клиенту связать ответ с запросом если он потерял `job_id`.
- `_force_provider_id` — внутренний флаг, позволяющий принудительно выбрать LLM-провайдера для конкретного job. Уже есть.

## Дедупликация и идемпотентность

Сейчас:
- **Idempotency-Key header** — клиент шлёт уникальный ключ (например sha256 содержимого), повторный POST с тем же ключом возвращает первый созданный job.

Что добавить:
- **Auto-dedup по hash файла** (опционально через env флаг). Если включён — система при создании job считает sha256 файла и проверяет нет ли уже job с тем же hash в этой организации за последние N дней. Если есть — возвращает существующий job_id вместо создания нового. Полезно для случая «бухгалтер случайно прислал тот же счёт повторно».

## Batch операции

### Способ 1: Zip-архив
`POST /api/v1/jobs/batch` принимает zip. Внутри — PDF/JPG + опционально `manifest.json`:
```json
{
  "documents": [
    {"file": "invoice-001.pdf", "document_hint": "invoice", "external_id": "INV-001"},
    {"file": "ttn-042.pdf", "document_hint": "TTN", "external_id": "TTN-042"}
  ]
}
```
Сервер распаковывает, создаёт N jobs, возвращает массив `job_id`.

### Способ 2: Multi-multipart
N полей `file_1`, `file_2`, `file_N` + общие настройки. Менее структурированно, но проще для curl-скриптов.

### Способ 3: Stream через SFTP
Просто положить пачку в папку — sweeper подхватит каждый отдельно.

**Срок реализации:** zip-вариант — 1 день. Остальные — позже.

## Открытые вопросы

1. **Шифрование передаваемых данных** — TLS-only достаточно или нужно ещё app-level (PGP/JWE)? Зависит от чувствительности контента.
2. **Retention входящих файлов** — 30 дней default. Бухгалтерия хочет 5 лет? Юристы — навсегда? Нужно решение про политику.
3. **Какие external_id критичны** — связь с 1С через ГУИД? Через номер документа? Через batch-id?
4. **mTLS vs API key** — для интеграции с банками обычно mTLS. У нас банки в скоупе?
5. **Multitenancy для интеграторов** — один API ключ = одна организация, или один ключ может писать в разные orgs (как у нашего root API_KEY)?
