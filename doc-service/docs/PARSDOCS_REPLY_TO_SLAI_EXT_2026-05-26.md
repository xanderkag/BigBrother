## Фрейминг — это внутренний микросервис, не внешняя интеграция

Перекалибровались. SLAI и parsdocs — наши же микросервисы; «BYO LLM
credentials» — это не коммерческий вопрос («кто платит Anthropic»), а
**архитектурное удобство**: ключ физически живёт в SLAI (там уже AI-инфра),
parsdocs получает его в заголовке вместо своей настройки. Это правильный
паттерн для multi-consumer (после SLAI могут быть другие микросервисы —
парсдокс не должен про каждого знать).

Из 5 пушбэков отпадают/упрощаются:

- **#2 (rate-limit/billing)** — не актуально как тема. Один общий внутренний
  ключ, throttling парсдоксу не нужен, ошибки от Anthropic прозрачно
  проксируем через `error.code='llm_rate_limited'`.
- **#3 (proxy vs BYO long-term)** — снимается. Микросервисная архитектура
  есть ответ: parsdocs делает свою зону (OCR + классификация + extract по
  схеме), консьюмер шлёт credentials per-request. Архитектурная встреча
  через квартал не нужна.
- **#4 (multi-tenant LLM-ключи / EXT-C)** — `blocked-on-trigger` без
  даты. Один внутренний ключ покрывает обозримое будущее.

## Бэклог parsdocs (в нашей очереди)

| ID | Эпик | Оценка | Status |
|---|---|---|---|
| **Q10 / EXT-A** | `GET /capabilities` + alias `X-Extractor-Signature` для outbound webhook | ½ дня | OPEN → parsdocs |
| **Q11 / EXT-B** | BYO LLM credentials через `X-LLM-*` заголовки + redaction в логах + ENV-флаг + метрики `extractor_llm_*` | 1-2 дня | OPEN → parsdocs |
| **Q12 / EXT-D** | Pre-upload signed URL (приём файла по URL вместо multipart, снимает 50MB-bottleneck) | 1 день | OPEN → parsdocs, после A+B |

**Итого MVP = 2 дня (A+B).** D — следом, не блокер.

Открываем работу с EXT-A — даёт SLAI возможность писать contract-test
первым делом.
