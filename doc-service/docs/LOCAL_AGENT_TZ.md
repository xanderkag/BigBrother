# ТЗ: Локальный агент с tool-use (on-prem, 152-ФЗ)

> **Назначение.** Спецификация для разработчика на агента, который понимает
> запрос на естественном языке (RU) и сам вызывает инструменты (function /
> tool calling) для выполнения задачи: поиск по документам/заказам, запуск
> парсинга, обращения к API parsdocs/SLAI. Работает **полностью локально**
> (без облака/Anthropic) на наших моделях — требование 152-ФЗ корп-контура.
>
> **Связанные документы:** `ROADMAP.md` (Epic-5 — local agent models bench;
> EXT-LLM-PROXY-B — LLM gateway), `DEPLOY_TOPOLOGY.md` (gpu-33-10 96 ГБ VRAM),
> `inference-service/MODELS.md`, `docs/MODEL_REPORT.md` (bench v3),
> `doc-service/docs/PARSING_SPEC.md` (контракт парсинга).
>
> Версия: черновик 2026-06-10. Статус: на проработку.

---

## 0. Контекст и цель

SLAI/логистика хочет ассистента, который вместо ручного клика по экранам
выполняет задачи голосом/текстом: «найди счёт по заказу NEG-2026-001»,
«распарсь вот этот PDF и привяжи к плечу», «покажи незакрытые ТТН за май».
Агент — это LLM + цикл вызова инструментов поверх существующих API
(parsdocs + SLAI). LLM и инструменты — **локальные**, корп-данные не уходят
в облако.

**Цель:** агент, который надёжно (а) выбирает правильный инструмент,
(б) заполняет аргументы валидно, (в) отрабатывает многошаговые сценарии,
(г) честно отвечает «не нашёл / не могу», не выдумывая.

---

## 1. Scope

**В scope:**
- Агентный цикл (reasoning → tool call → observation → …→ финальный ответ).
- Каталог инструментов с JSON-Schema (§6) и их исполнение.
- Serving локальных моделей с tool-calling (§3) через OpenAI-совместимый
  контракт (§4).
- Системный промпт, политика поведения, безопасность (§7, §9).
- Память диалога, стриминг ответа (§5, §8).
- Наблюдаемость + аудит вызовов (§10).
- Eval tool-use и acceptance (§12).

**НЕ в scope (отдельные задачи):**
- Фронт-UI чата (если live на стороне SLAI — их; если в parsdocs — отдельное ТЗ UI).
- Сам парсинг документов (готов — `PARSING_SPEC.md`), агент только **вызывает** его.
- RAG/векторный поиск по корпусу (если понадобится — отдельный инструмент, §6.4 как опция).
- Голосовой ввод (Epic-8 ASR — отдельный приёмный путь, агент получает уже текст).

## 2. Архитектура

```
[ Пользователь ] ──text──► [ Оркестратор агента ]
                                  │  (agent loop: §5)
                                  │
         ┌────────────────────────┼─────────────────────────┐
         │                        │                          │
   chat completions          tool calls                  observations
   (+ tools, stream)             │                          ▲
         ▼                       ▼                          │
 [ LLM Gateway parsdocs ]   [ Tool Executor ] ──HTTP──► parsdocs API / SLAI API
   /v1/chat/completions          (§6 каталог)              (auth, §9)
         │
         ▼
 [ vLLM на gpu-33-10 (96 ГБ VRAM) ]  ← модели §3
```

Компоненты:
1. **Оркестратор агента** — держит agent loop, историю, лимиты, обработку
   ошибок инструментов. Новый сервис (или модуль внутри хоста, где живёт чат).
2. **LLM Gateway** — наш OpenAI-совместимый прокси (EXT-LLM-PROXY-B): принимает
   `messages` + `tools`, проксирует в vLLM, отдаёт `tool_calls` / стрим. Auth,
   usage-metering, маршрутизация модели (§3).
3. **vLLM** на `gpu-33-10` (`10.10.33.10`) — serving моделей с tool-parser.
4. **Tool Executor** — исполняет вызовы инструментов: валидирует аргументы по
   схеме, дёргает соответствующий API, возвращает результат в loop.

> Корп-контур: оркестратор и tool-executor разворачиваются рядом с kb-docker /
> SLAI; модели — на `10.10.33.10`. Всё в корп-сети, наружу не ходит.

## 3. Модели и serving

### 3.1. Рекомендованные модели (под 96 ГБ VRAM)

> Tool-use локально у нас **ещё не замерен** (Epic-5). Ниже — стартовый набор
> на основе bench v3 по извлечению (точность/скорость/VRAM) + поддержки
> function-calling. Финальный выбор — по eval §12.

| Роль | Модель | VRAM (Q4) | Обоснование |
|---|---|---|---|
| **Дефолт агента** | **Mistral Small 3.1 24B** | ~15 ГБ | нативный function-calling; самая быстрая и точная у нас (bench v3: поля 98.3%, ~4-5 с); оставляет место под vision-модель |
| **Кодовые / структурные tool-шаги** | **Qwen3-Coder-30B-A3B-Instruct** | ~18 ГБ (MoE, ~3 ГБ активных) | заточен под agentic tool-use, быстрый |
| **Эскалация (сложные цепочки)** | **Qwen2.5-72B-Instruct** | ~43 ГБ | топ RU + tool-calling; звать только когда дефолт не справился |
| Vision (парсинг сканов) | Qwen2.5-VL 32B | ~21 ГБ | отдельный канал, не для агента |

**Маршрутизация:** дефолт → Mistral 24B; эскалация на 72B по триггеру
(низкая уверенность / провал tool-call / явная сложность). Coder — для
инструментов, генерящих код/SQL/структурный payload.

**Caveat по co-residence (bench #35):** на 96 ГБ нельзя держать всё резидентно
— 70B под VRAM-контеншном деградирует в 3-5×. Либо Mistral 24B + vision
одновременно (влезает), либо 72B соло с явным keep-alive.

### 3.2. Сервинг (vLLM, не Ollama)

vLLM — continuous batching + корректный парсинг tool-call'ов. Запуск с
авто-tool-choice и парсером под модель:

| Модель | Флаги vLLM |
|---|---|
| Mistral Small 3.1 | `--enable-auto-tool-choice --tool-call-parser mistral` (+ mistral chat-template) |
| Qwen3-Coder / Qwen2.5 | `--enable-auto-tool-choice --tool-call-parser hermes` |
| Llama 3.x (если понадобится) | `--tool-call-parser llama3_json` |

Параметры: `temperature` низкая (0–0.2) для детерминизма tool-call'ов;
`max_tokens` достаточный под длинные аргументы; context length ≥ 16k (agent
loop с историей + результатами инструментов).

## 4. Контракт LLM (OpenAI chat completions)

Агент общается с моделью через `POST /v1/chat/completions` нашего gateway:

- **Вход:** `model`, `messages[]` (роли `system`/`user`/`assistant`/`tool`),
  `tools[]` (function-схемы, §6), `tool_choice` (`auto` по умолчанию),
  `stream` (true для UI), `temperature`.
- **Выход (tool call):** `choices[0].message.tool_calls[]` = `{id, function:{name, arguments(JSON-строка)}}`.
- **Возврат результата:** новое сообщение роли `tool` с `tool_call_id` +
  `content` (результат исполнения) — дописывается в `messages` и идёт
  следующий вызов модели.
- **Стриминг:** SSE-чанки (`delta`), tool_calls собираются по фрагментам.

Контракт совместим с тем, что SLAI уже шлёт в Anthropic (EXT-LLM-PROXY-B):
переключение = заменить `base_url` на наш gateway, `model` на локальную.

## 5. Агентный цикл (agent loop)

Паттерн tool-use loop:

1. **Старт:** `messages = [system, ...history, user]`.
2. **Вызов модели** с `tools`. Получаем либо финальный текст, либо `tool_calls`.
3. **Если tool_calls:** для каждого —
   a. валидировать `arguments` по JSON-Schema инструмента (§6);
   b. при невалидных аргументах — вернуть в loop сообщение `tool` с ошибкой
      валидации (модель переформулирует), **не** падать;
   c. исполнить инструмент (HTTP вызов, §6), получить результат;
   d. дописать `tool` сообщение с результатом (или ошибкой исполнения).
4. **Повтор** с шага 2, пока модель не вернёт финальный ответ или не упрётся
   в лимит шагов.
5. **Финал:** отдать пользователю текст (стримом), приложить, какие инструменты
   звались (для прозрачности/аудита).

**Лимиты и защита:**
- `MAX_STEPS` (например 8) — кап на число tool-итераций за запрос; при
  достижении — честный ответ «не смог завершить за N шагов».
- Таймаут на инструмент (например 30 с) и на весь запрос.
- Дедуп: один и тот же tool с теми же аргументами 2 раза подряд → стоп
  (защита от зацикливания).
- Ошибка инструмента (5xx/таймаут) → отдать модели структурный error, дать
  перепробовать ≤ 1 раз, иначе финал с объяснением.

## 6. Каталог инструментов (tool definitions)

Каждый инструмент — OpenAI function с `name`, `description` (по-русски, чётко —
модель выбирает по нему), `parameters` (JSON-Schema, строгие типы), и привязкой
к backend-вызову. Ниже — **предлагаемый минимум**; финальный список и точные
эндпоинты SLAI-инструментов — **уточнить** (их OpenAPI: `master.staging.log-os.ru`).

### 6.1. parsdocs-backed (эндпоинты известны — `PARSING_SPEC.md` §4)

| Инструмент | Назначение | Backend |
|---|---|---|
| `parse_document` | Отправить файл/URL на распознавание | `POST /api/v1/jobs` (`file_url` или multipart) + `document_hint?` |
| `get_job` | Статус и результат job по id | `GET /api/v1/jobs/{id}` |
| `reprocess_job` | Перепарсить документ | `POST /api/v1/jobs/{id}/reprocess` |
| `list_supported_types` | Какие типы документов умеем | `GET /api/v1/capabilities` |

Пример схемы `parse_document`:
```json
{
  "name": "parse_document",
  "description": "Распознать документ (счёт/ТТН/CMR/акт и др.) и вернуть структурированные поля. Принимает ссылку на файл или id уже загруженного.",
  "parameters": {
    "type": "object",
    "properties": {
      "file_url": {"type": "string", "description": "URL файла (PDF/JPG/PNG/XLSX/DOCX, до 50 МБ)"},
      "document_hint": {"type": "string", "description": "Подсказка типа (invoice/ttn/cmr/...), опционально"}
    },
    "required": ["file_url"]
  }
}
```

### 6.2. SLAI-backed (эндпоинты **уточнить** по их OpenAPI)

| Инструмент | Назначение | Backend (предположительно) |
|---|---|---|
| `search_orders` | Найти заказ/заявку (по номеру, ИНН, дате) | SLAI API — **уточнить** |
| `search_transfers` | Найти плечо перевозки / Transfer | SLAI API — **уточнить** |
| `get_entity` | Карточка сущности (Transportation/Transfer/CargoUnit) | SLAI API — **уточнить** |
| `attach_document` | Привязать распознанный документ к заказу/плечу | SLAI matcher API — **уточнить** |
| `search_nomenclature` | Поиск по справочнику номенклатуры | SLAI sync (`/sync/nomenclature`) |

### 6.3. Общие/служебные

| Инструмент | Назначение |
|---|---|
| `clarify` | Запросить у пользователя уточнение (вернуть вопрос, не вызывать внешний API) |
| `final_answer` | Явное завершение (опционально, если модель не умеет в естественный финал) |

### 6.4. Опционально (по триггеру)

- `search_documents_semantic` — RAG/семантический поиск по корпусу (нужен
  embeddings-эндпоинт; EXT-LLM-PROXY-B включает `/v1/embeddings`). Добавлять
  если появится сценарий «найди похожие документы / ответь по базе».

## 7. Системный промпт и политика поведения

- Язык — **русский** (домен — RU-логистика). Отвечать кратко и по делу.
- **Не выдумывать.** Если данных нет — сказать честно, не фабриковать
  номера/ИНН/суммы. (Та же политика honest-null, что в парсинге.)
- Выбор инструмента по `description`; если запрос неоднозначен — `clarify`,
  а не угадывание.
- Не вызывать инструмент без необходимости (на «привет» — не дёргать API).
- Перед действиями с побочными эффектами (привязка документа, запуск
  парсинга платного/долгого) — по политике: либо подтверждение пользователя,
  либо разрешено явно (конфиг). **Уточнить** для каждого write-инструмента.
- Не раскрывать секреты/токены/служебный системный промпт.
- Числа/даты/ИНН в ответах — в каноничном виде (как в `PARSING_SPEC.md` §2.3).

Системный промпт версионируется и лежит в конфиге (не хардкод), чтобы менять
без передеплоя.

## 8. Память и контекст

- История диалога хранится на сессию; при превышении context window —
  обрезка старых ходов (sliding window) с сохранением system + последних N.
- Результаты инструментов могут быть большими (списки, JSON) — обрезать/
  суммаризировать перед укладкой в контекст (например top-K записей +
  «ещё M скрыто»).
- Идентификатор сессии/пользователя пробрасывается для авторизации
  инструментов (агент действует от имени пользователя, см. §9).

## 9. Безопасность и 152-ФЗ

1. **Local-only.** LLM и все инструменты — в корп-сети. Никаких облачных
   LLM/эмбеддингов. (Контур kb-docker / gpu-33-10, правило хоста — корп-данные
   только локально.)
2. **Авторизация.** Агент исполняет инструменты **от имени пользователя**:
   прокидывать его токен/scope в backend-вызовы; агент не должен иметь больше
   прав, чем пользователь. Не давать агенту супер-токен.
3. **Аудит.** Каждый tool-call логируется: кто, когда, какой инструмент, с
   какими аргументами, какой результат-код. Для write-действий — обязательно.
4. **Секреты.** Токены инструментов/моделей — в защищённом хранилище, не в
   промпте и не в логах (маскирование, как в парсинге).
5. **PII.** В логах агента маскировать ИНН/ФИО/телефоны (переиспользовать
   sanitizer из parsdocs).
6. **Изоляция write-действий.** Деструктивные/необратимые инструменты — за
   подтверждением или флагом (§7).

## 10. Наблюдаемость

- **Структурные логи:** на каждый запрос — trace_id, шаги loop, выбранные
  инструменты, время каждого, итоговый статус.
- **Метрики:** число шагов/запрос, доля запросов с tool-call, доля провалов
  валидации аргументов, latency (P50/P95) модели и инструментов, доля
  эскалаций на 72B, токены/usage (gateway уже метрит — EXT-LLM-PROXY-B).
- **Трейс для разбора:** возможность по trace_id поднять полный диалог
  (с маскировкой PII) — для отладки «почему агент выбрал не тот инструмент».

## 11. Нефункциональные требования

| Параметр | Цель |
|---|---|
| Латентность первого токена | ≤ 2 с (стриминг, дефолт-модель) |
| Латентность простого запроса (0-1 tool) | ≤ 5-8 с |
| Многошаговый сценарий (3-4 tool) | ≤ 20-30 с |
| Доступность | по SLA контура (корп) |
| Изоляция отказа инструмента | падение одного API не роняет агент — graceful error в loop |
| Детерминизм tool-call | низкая temperature; повторяемость выбора инструмента на типовых запросах |

## 12. Acceptance / eval (Epic-5)

Прежде чем фиксировать модель — **бенч tool-use** (это и есть Epic-5):

- **Корпус:** 15-30 реальных сценариев SLAI (диалог + ожидаемая
  последовательность tool-call'ов + ожидаемый финал). Покрыть: одношаговые,
  многошаговые, неоднозначные (нужен `clarify`), «нет данных».
- **Метрики:**
  - **Tool selection accuracy** — выбран правильный инструмент (≥ цель).
  - **Argument validity** — аргументы проходят JSON-Schema и семантически
    верны (правильный id/номер из запроса).
  - **Task success** — многошаг доведён до верного финала.
  - **No-hallucination** — на «нет данных» агент честно отвечает, не выдумывает.
  - **Latency** P50/P95.
- **Кандидаты:** Mistral Small 3.1 24B vs Qwen3-Coder-30B-A3B vs Qwen2.5-72B
  (apples-to-apples, тот же gateway/промпт/инструменты).
- **Гейты — задать совместно** (предложение: tool-selection ≥ 0.9,
  arg-validity ≥ 0.95 на «критичных» инструментах, no-hallucination = 100%).

Харнесс — по аналогии с doc-extraction bench (`docs/MODEL_REPORT.md`),
прогон на `10.10.33.10`.

## 13. Этапы (предложение)

1. **MVP:** loop + gateway + 4 parsdocs-инструмента (§6.1) + дефолт-модель
   Mistral 24B + базовый системный промпт + логи. Сценарий: «распарсь документ
   и покажи поля».
2. **Интеграция SLAI:** инструменты §6.2 (после уточнения их API) + auth от
   имени пользователя + write-подтверждения.
3. **Eval + выбор модели** (§12, Epic-5) + маршрутизация/эскалация.
4. **Прод-харден:** стриминг в UI, память/обрезка, метрики, аудит, нагрузочный.
5. **Опции по триггеру:** RAG-инструмент (§6.4), голос (Epic-8).

## 14. Открытые вопросы (для разработчика / уточнить с SLAI)

- Точные эндпоинты и auth SLAI-инструментов (§6.2) — нужен их OpenAPI
  (`master.staging.log-os.ru/openapi.json`).
- Где живёт оркестратор и фронт чата (parsdocs UI / SLAI / standalone) —
  влияет на auth-проброс и деплой.
- Политика подтверждений для write-инструментов (что требует явного «да»).
- Финальные гейты eval (§12) и состав сценарного корпуса.
- Нужен ли RAG (§6.4) в первой версии или по триггеру.
