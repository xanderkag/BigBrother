# parsdocs UI — UX-анализ

**Дата:** 2026-06-01 (файл от 2026-05-31)
**Объём:** SPA `doc-service/ui` (React + TS + Tailwind + React Query + react-pdf).
13 страниц, 13 компонентов. Прод: `https://parsedocs.taipit.ru/ui/`.
**Метод:** статический разбор исходников (роутинг, Layout, 5 ключевых
экранов, компоненты данных). Не запускали в браузере — поэтому findings
по перформансу/анимациям помечены как «гипотеза». Каждый пункт — со
ссылкой на файл, чтобы можно было проверить и поправить.

> **Кто пользователь.** Оператор бухгалтерии/логистики, который весь день
> разгребает очередь распознанных документов: грузит пачку, смотрит что
> система не уверена, исправляет, одобряет → дальше webhook в учётную
> систему. Главная метрика UX здесь — **сколько секунд и кликов на один
> документ в режиме «проверка-правка-одобрение»**, и **насколько быстро
> глаз ловит, что именно не так**. Всё остальное (админка типов,
> провайдеры, тенанты) — вторично.

## 1. Информационная архитектура и навигация

**Структура (`Layout.tsx`, `lib/sidebar.ts`):** свёртываемый левый сайдбар
(240/60 px, состояние в `localStorage`, тоггл по `Ctrl/Cmd+B`), топбар с
хлебными крошками, поиском, индикатором статуса системы, переключателем
темы и ролью. Разделы:

- **Основное:** Dashboard · Журнал работ · Очередь ревью · Upload · Тест. лаборатория
- **Админ:** Document Types · Providers · Audit Log · Tenants · Reference Lists · Settings

**Хорошо:**
- Живые счётчики у пунктов (jobs / review / типы / провайдеры) — оператор
  сразу видит размер очереди.
- Свёрнутый режим + хоткей + сохранение состояния — забота о плотности.
- Активный пункт с индиго-акцентом, индикатор статуса системы в топбаре.

**Проблемы:**
- **F4. Лендинг — Dashboard, а не очередь.** Index-роут `/` → `DashboardPage`
  (`App.tsx`). Для оператора, чья работа — ревью, это лишний клик каждый
  раз. → Сделать лендинг ролезависимым (оператор → `/review`, админ →
  Dashboard) либо вынести «Открыть очередь» крупной кнопкой на Dashboard.
- **F12. Смесь языков** в одном меню («Журнал работ» рядом с «Document
  Types», «Upload», «Audit Log»). Мелочь, но бьёт по ощущению
  законченности. → Один язык интерфейса (русский для операторских,
  термины оставить только где нет хорошего перевода).
- **Свёрнутый сайдбар теряет число** счётчика (остаётся точка) — а именно
  число (сколько в очереди) самое ценное. → Бейдж-число поверх иконки.
- Инициалы пользователя = первые 2 символа hex UUID (в коде честно
  помечено «не идеально»). Косметика, но видна в футере каждый экран.

## 2. Сквозной сценарий оператора

### 2.1 Загрузка (`Upload.tsx`)

**Хорошо:** drag-drop + пикер, мультизагрузка с семафором `MAX_PARALLEL=3`,
постатусная очередь (ждут/грузятся/готово/ошибок), история последних 20
(localStorage), выбор `document_hint` и `webhook_url`.

**Проблемы:**
- Опции (`document_hint`, `webhook_url`) — **на всю пачку, не пофайлово**,
  хотя UX мультизагрузочный. Для смешанной пачки (счёт + ТТН) подсказку
  типа задать нельзя. → Либо честно «авто-классификация» и убрать hint,
  либо пофайловый выбор.
- Не экспонированы `metadata` и `redact_pii`, хотя API их принимает.
  `redact_pii` особенно важен для PII-чувствительных документов — стоит
  вынести чекбоксом.
- **Автопереход после одиночной загрузки — через `setTimeout(…,100)`** и
  перечитывание localStorage (в коде помечено «hack»). Хрупко: при
  медленном ответе редирект уедет не туда. → Брать `id` из ответа API
  напрямую.
- Нет прогресс-бара пофайлово (только текст-статус) и нет отмены после
  старта. Для 50-МБ сканов прогресс заметно снижает тревожность.
- Клиент не валидирует MIME/размер (`ACCEPT_MIME` — мёртвый экспорт,
  валидация только на бэке). → Лёгкая клиентская проверка экономит круг
  до сервера и даёт мгновенный фидбек.

### 2.2 Список работ (`JobsList.tsx`)

**Хорошо:** богатая таблица (тип + `TierBadge`, статус + webhook-чип,
`ConfidenceBar`, число issues, длительность, модель/OCR, возраст);
табы-фильтры со счётчиками; фильтры по происхождению (реальные/синтетика)
и типу; состояние в URL (шарится ссылкой); адаптив в карточки на мобиле;
скелетон при загрузке, отдельный `EmptyState` (различает «нет док» vs «нет
по фильтру»), баннер ошибки. Bulk-бар (Одобрить / Перепрогнать).

**Проблемы:**
- **F7. Кликабельно только имя файла** (`<Link>`), остальная строка — нет.
  Маленькая мишень для целодневной работы. → Кликабельна вся строка.
- **F6 (часть).** Bulk-одобрить можно дёрнуть и на не-`needs_review` —
  узнаёшь об отказе только из `alert`. → Прятать/дизейблить «Одобрить» для
  неподходящих статусов в выборке.
- Docstring обещает «Auto-refresh 10s», но `refetchInterval` стоит только
  на счётчиках (15 с), сам список обновляется кнопкой «↻». → Привести в
  соответствие; авто-рефетч только при наличии pending/processing (чтобы
  idle-вкладка не молотила сеть).
- Выбор строк переживает смену страницы/фильтра — задумано, но может
  удивлять при bulk-операции «вслепую».

### 2.3 Очередь ревью (`ReviewQueue.tsx`) — ключевой экран

**Хорошо:** группировка по типу (`<details>`), шапка со статистикой
(средняя уверенность с меткой низкая/средняя/высокая, топ-issue), фильтр
по **категории проблемы** (ИНН/КПП, Суммы, Даты, Другое — эвристика
`classifyIssue`), превью важных полей с амбер-подсветкой проблемных/
низкоуверенных, список issues (первые 3 + «ещё N»).

**Проблемы:**
- **F1 (🔴, подтверждено в коде).** `PREVIEW_FIELDS` (`ReviewQueue.tsx:591`)
  = `seller_name`, `buyer_name`, `document_number`, `document_date`,
  читаются плоско `job.extracted[key]` (`:731`). Но реальная схема —
  **вложенная** (`ExtractedDataPanel` берёт `seller.name`, `number`/
  `doc_number`, `date`). → `job.extracted['seller_name']` = `undefined` →
  строка возвращает `null` (`:732`) → **превью «кто/кому/сколько/когда»
  пустое для обычных документов.** Самый важный скан-блок на самом важном
  экране не работает. Фикс — часы: брать `seller?.name ?? seller_name`,
  `number ?? document_number` и т.п. (или общий хелпер-флэттенер; рядом уже
  есть `lib/extracted-summary.ts`).
- **F5 (часть). Нет клавиатуры.** Ни j/k между карточками, ни хоткея
  «Одобрить», ни «открыть/перепрогон». Целодневная триажная работа без
  клавиатуры — это в разы медленнее. → j/k + Enter(открыть) + a(одобрить).
- **F6. Bulk-одобрение последовательное и рвётся на первой ошибке**
  (`break` + alert, без resume). Один битый job останавливает всю пачку,
  прогресс частичный и невозвратный. (В `JobsList` — наоборот
  `Promise.allSettled`; поведение между двумя экранами **непоследовательно**.)
  → Единый паттерн: продолжать, собрать ошибки, показать сводку в конце.
- Правка в очереди невозможна — только «Открыть» (уйти в деталь),
  «Перепрогон», «Одобрить». То есть любая правка = уход с экрана (см. F2/F8).

### 2.4 Детальная карточка + правка (`JobDetail.tsx`, `ExtractedEditor.tsx`)

**Хорошо:** двухколоночный лэйаут (PDF слева, данные справа) — правильная
модель для сверки. Sticky `ValidationBanner`, карточка обогащения
(DaData/ЕГРЮЛ + расхождения), `FieldConfidenceCard` (бары по полям,
отсортированы по возрастанию — сразу видно худшие), `WebhookDeliveryCard`,
сворачиваемые шаги пайплайна, поддержка multi-doc (табы) и classify-only.

**Проблемы:**
- **F2 (🔴). Правка — это сырой JSON в `<textarea>`** (`ExtractedEditor`).
  Для бухгалтера/логиста это нерабочий интерфейс: легко сломать структуру,
  нет подсказок полей, нет валидации по схеме на лету. В коде честно
  отложено в «Phase 4». → Структурный per-field редактор (форма из той же
  схемы, что рисует `ExtractedDataPanel` в режиме Form). Это самый крупный,
  но и самый ценный UX-долг.
- **F3 (🟠). Нет dirty-guard.** Клик по backdrop закрывает редактор; при
  непустом тексте — потеря правок без предупреждения. → `beforeunload`/
  confirm при наличии изменений; не закрывать по фону, только по крестику/Esc.
- **F8. Нет «пред/след документ».** Правка пачки = открыл → поправил →
  назад в список/очередь → открыл следующий. → Стрелки ←/→ (и хоткеи)
  внутри детального экрана, идущие по текущему отфильтрованному списку.
- **F11. Нет «скачать оригинал», «скачать JSON», «переотправить webhook».**
  Доставка webhook видна (`WebhookDeliveryCard`), но retrigger из UI нет —
  при сбое доставки оператор бессилен. → Кнопка «Переотправить»; экспорт
  JSON/CSV; «скачать исходник».
- Уверенность в шапке — **процент-текст** «via {ocr_engine}», а в остальном
  UI это сканируемый цветной бар. Непоследовательно. → Тот же `ConfidenceBar`
  и в шапке.
- Загрузка детали — центрированный текст «Загрузка job'а…», без скелетона
  (в списке скелетон есть). Мелкая непоследовательность.

## 3. Компоненты отображения данных

- **`ConfidenceBar`** — хорошо: бар + процент (бар `aria-hidden`, значение
  несёт текст → доступно), пороги ≥0.85 emerald / ≥0.6 amber / <0.6 rose,
  клампинг 0..1, null→«—». **Минус:** пороги продублированы в трёх местах
  (`ConfidenceBar`, `FieldConfidenceCard`, `ExtractedDataPanel.Field`) →
  риск рассинхрона. → Вынести в общий конфиг.
- **`ExtractedDataPanel`** — Form/JSON-тоггл (сохраняется), Copy (весь JSON),
  секции Реквизиты/Продавец/Покупатель/Позиции/Флаги/Прочее, амбер-/роза-фон
  по уверенности, подсветка по упоминанию в issue. Добротно. Form-режим
  только для чтения — отсюда и боль F2.
- **`ValidationBanner`** — sticky, список **сырых текстовых** issue-строк,
  не привязаны к полям. → В идеале клик по issue скроллит/подсвечивает поле.
- **`PdfViewer`** — react-pdf, fit-by-width через `ResizeObserver`, зум
  0.5–3×, перелистывание (если >1 стр.), фоллбэк `<img>` для картинок.
  **Минусы:** нет клавиш-перелистывания, нет fit-to-width/поворота/поиска
  по тексту; зум сбрасывается на resize. Для сверки скана важен поиск/зум.
- **`TierBadge`** — stable/beta/exp с тултипом. (Опечатка в строке тултипа:
  «valиdators» — кириллическая «и».)

## 4. Сквозные наблюдения

- **F9 (безопасность UX).** `RequireAuth` — бинарная проверка токена, **без
  ролевого гейта**. Админ-роуты (`/tenants`, `/providers`, `/settings`)
  достижимы любым залогиненным по прямому URL. UI это не запрещает (бэк
  может — но интерфейс не должен показывать недоступное). → Ролевые guard'ы
  на роутах + скрытие пунктов меню.
- **F10.** Все подтверждения/ошибки — нативные `window.confirm`/`alert`,
  включая действия с **внешним эффектом** (одобрение → webhook-доставка).
  Это и некрасиво, и легко прокликивается. → In-app модалка с явным
  описанием последствий (особенно «это разошлёт webhook в учётные системы»).
- **Непоследовательность bulk-логики** между `JobsList` (allSettled) и
  `ReviewQueue` (sequential + break). Выбрать один контракт.
- **Доступность:** бар уверенности правильно `aria-hidden` + текст; но
  отсутствие клавиатурной навигации в ревью/PDF и тоггл-зоны сайдбара
  (`aria-hidden`, `tabIndex=-1`) — частичная клавиатурная история.

## 5. Рекомендованный порядок (по соотношению ценность/усилие)

**Спринт «быстрые победы» (≈2-3 дня суммарно):**
1. **F1** — починить ключи превью в очереди ревью (часы, но чинит главный
   экран). ⬅ начинать с этого.
2. **F3** — dirty-guard в редакторе (защита от потери правок).
3. **F4** — ролезависимый лендинг (оператор → `/review`).
4. **F7** — кликабельна вся строка списка.
5. **F12** — единый язык навигации; **F11** — кнопка «скачать JSON» (тривиально).

**Средний горизонт (≈неделя):**
6. **F5** — клавиатурная триаж-навигация в ревью + перелистывание PDF.
7. **F8** — пред/след документ на детальном экране.
8. **F6** — единый bulk-контракт (продолжать + сводка ошибок).
9. **F9** — ролевые guard'ы на роуты.
10. **F10** — заменить native confirm/alert на in-app модалку для
    действий с внешним эффектом.

**Крупная ставка (отдельный спринт):**
11. **F2** — структурный per-field редактор извлечённых данных вместо сырого
    JSON. Самый большой выигрыш для целевого пользователя; делать на базе
    уже существующего Form-рендера `ExtractedDataPanel`.

## 6. Оговорки

- Анализ статический: реальные тайминги, анимации, поведение на больших
  очередях и на мобиле в браузере не проверялись — это гипотезы по коду.
- Не проверялись: `Dashboard`, `DocumentTypes`, `Providers`, `Tenants`,
  `Settings`, `AuditLog`, `ReferenceLists`, `TestLab`, `Login` (админ/
  служебные — вторичны для операторского UX; при необходимости разберём
  отдельно).
- Findings с указанием `файл:строка` проверены в коде (F1, F2, F3, F4, F9);
  остальные — из карты экранов, перепроверить точечно перед фиксом.
