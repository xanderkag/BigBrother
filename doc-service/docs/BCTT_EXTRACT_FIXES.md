> **Вход:** отчёт SLAI «Ванге: точность экстракции» от 2026-07-16 (корпус `bctt_v2`, 138 док,
> улики по `origin_file`). **Здесь:** корневая причина по коду + правки. Улики SLAI подтверждены,
> механизм найден и он **один на 3 из 4 находок**.
> Сверка: [BCTT_GROUNDTRUTH.md](./BCTT_GROUNDTRUTH.md) · предыдущий цикл: [BCTT_EVAL_FIXES.md](./BCTT_EVAL_FIXES.md)

## 🔴 КОРЕНЬ: регистр слага = промах мимо типизированного парсера

SLAI заметил: «регистр типа коррелирует с качеством **стопроцентно**» — 3 док с типом `CMR`
(заглавные) имеют маршрут/стороны/машину/ТНВЭД, все 8 с `cmr` (строчные) — **без маршрута вообще**.

Это не косметика. Это буквально баг диспетчеризации парсеров:

```ts
// src/types/documents.ts:19
export const DOCUMENT_TYPES = ['invoice', 'factInvoice', 'UPD', 'TTN', 'CMR', 'AKT'] as const;

// src/types/documents.ts:29  ← регистрозависимо!
export function isBuiltinDocumentType(slug: string): slug is BuiltinDocumentType {
  return (DOCUMENT_TYPES as readonly string[]).includes(slug);
}

// src/pipeline/parsers/index.ts:62
get(slug: DocumentTypeSlug): DocumentParser {
  if (isBuiltinDocumentType(slug)) return this.builtins[slug];  // 'CMR' → CmrParser
  return this.getGeneric(slug);                                 // 'cmr' → GenericLlmParser
}
```

Две ветки извлечения одного и того же документа:

| Ветка | Кто ставит слаг | Слаг | Парсер | Схема | Итог |
|---|---|---|---|---|---|
| **Одиночный док** | keyword-классификатор `classifier/keywords.ts:80` → `{ type: 'CMR' }` | `CMR` | `CmrParser` (`parsers/cmr.ts:14`) | `DOCUMENT_JSON_SCHEMAS.CMR` — **в коде** | маршрут, стороны, машина, ТНВЭД ✅ |
| **Сегмент композита** | `multidoc/boundaries.ts:80` → `{ slug: 'cmr' }` | `cmr` | `GenericLlmParser` | из БД-оверрайда | ❌ пусто |

**Добивающая деталь (проверено на asha, `document_types`):** у строки `CMR` в БД
**`llm_schema = NULL`** — схемы legacy-builtin'ов живут в **коде**, а не в БД:

| slug | is_builtin | parser_kind | llm_schema |
|---|---|---|---|
| `CMR` / `TTN` / `AKT` / `factInvoice` | true | `llm_extract` | **NULL** ← схема в коде |
| `UPD` / `invoice` | true | `builtin:*_regex` | **NULL** ← схема в коде |
| `commercial_invoice` / `packing_list` / `contract_specification` | true | `llm_extract` | set |
| `vehicle_registration` / `customs_export_ead` | false | `llm_extract` | set |

Резолвер (`document-type-resolver.ts:206` `expandSlugCandidates`) регистр **терпит** — по `cmr` он
находит строку `CMR`. Но брать оттуда нечего: `llm_schema = NULL` → `GenericLlmParser` уходит в LLM
**вообще без CMR-схемы**.

**Следствие:** модель про `loading_place` / `delivery_place` / `border_crossing` **никогда не
спрашивали**. Поэтому их нет ни у одного из 8. И поэтому `number` принимает мусор («CMR», имя
перевозчика) — поле ничем не заякорено.

**Радиус больше CMR.** Те же грабли у **`ttn`, `upd`, `akt`, `tax_invoice`** — все legacy-builtin'ы
записаны в верхнем/camelCase регистре и имеют `llm_schema = NULL`. `invoice` **не задет** (builtin-слаг
сам в нижнем регистре) — ровно поэтому инвойсы в композитах извлекаются нормально, а CMR нет.

## ⚠️ Просьба SLAI №3, выполненная в лоб, ЗАМАСКИРУЕТ баг

SLAI просит «одно написание типа (`cmr`, не `CMR`)». Нормализатор уже есть и уже это делает —
`slug-normalize.ts:45` `normalizeSlugForApi('CMR') → 'cmr'`, но он **outbound-only**
(`jobsRepo.toApi()` + webhook payload; см. шапку файла: «БД и внутренний код продолжают использовать
историческое имя слага»).

Если просто дотянуть нормализацию на оставшийся путь — **обе ветки станут выглядеть одинаково
(`cmr`), а 8 из 11 останутся без маршрута**, и различить их снаружи будет уже нечем. Регистр сейчас —
единственный внешний признак, по которому SLAI этот баг и нашёл.

**Порядок правок обязателен: сначала FIX-A (диспетчеризация), только потом FIX-C (единое написание).**

## Правки

### FIX-A (P0) — канонизировать слаг перед builtin-проверкой
**Чинит:** отсутствие маршрута у 8/11 CMR + мусор в `number` (главный приоритет SLAI №2).

Ввести inbound-канонизацию (обратную `OUTBOUND_SLUG_ALIASES`) в **одной точке** — `ParsersFactory.get()`:
`cmr→CMR`, `ttn→TTN`, `upd→UPD`, `services_act→AKT`, `tax_invoice→factInvoice`; затем builtin-проверка.

- Файлы: `src/types/slug-normalize.ts` (добавить `canonicalizeSlugForParser` + INBOUND-мапу как
  инверсию существующей), `src/pipeline/parsers/index.ts:62`.
- Альтернатива (хуже): продублировать схемы в БД (`llm_schema` для CMR/TTN/UPD/AKT) — даёт дрейф
  двух источников правды. **Не рекомендую.**
- Не трогать `isBuiltinDocumentType` (это type-guard, от него зависит типизация `builtins[slug]`) —
  канонизировать **до** вызова.

### FIX-B (P0) — ИНН берётся из платёжных реквизитов
**Улики:** заказ #5, `inv_1.jpeg` + `pac_1.jpeg` → `seller.inn = 7707083893` = **ПАО «Сбербанк»**,
при том что продавец по `CMR.jpeg` — `SIA BALTEREX` (Латвия, росс. ИНН невозможен в принципе).

**Почему дорого:** оба документа дают **одно и то же** значение → кросс-документная сверка SLAI
считает поле «сошедшимся» и красит зелёным. Контрольная сумма ИНН проходит, ЕГРЮЛ проходит — ИНН
реальный, просто **чужой**. Систематическая ошибка маскируется под подтверждённую истину.

> ⚠️ **Исправление первой редакции (2026-07-17).** Здесь было заявлено, что чинить надо схему/промпт
> `commercial_invoice`/`packing_list`. **Неверно** — виноват детерминированный нормализатор.

**Причина (найдена по реальным данным на asha, не по догадке):** `recoverPartyInnsFromText` (F0,
`normalize/inn-recovery.ts`). Он ищет метку стороны (`поставщик`/`продавец`/…), затем **ПЕРВЫЙ**
`ИНН \d{10|12}` в окне 400 символов. Если в окно попадает блок платёжных реквизитов
(«Банк получателя: ПАО СБЕРБАНК · ИНН … · БИК … · Сч. №»), забирается **ИНН банка**. Он настоящий и
проходит checksum — поэтому гейт `normalizeInn` его пропускал.

**Улика, которая это доказала** (`jobs.extracted` на asha): в выводе есть `seller`/`buyer`, которых
**НЕТ в схемах** (`commercial_invoice` → `exporter`/`consignee`/`buyer`; `packing_list` → вообще без
`seller`/`buyer`), и **НЕТ `exporter`**, который в схеме **есть**. Модель так ответить не могла —
поля дописывал нормализатор, создавая party-объект с нуля (`partyObj = {}`). Схема/промпт ни при чём.

**Фикс:**
1. Окно стороны обрезать на первом банковском маркере (`банк`/`БИК`/`к/с`/`Сч. №`/`р/с`): собственный
   ИНН стороны стоит рядом с названием, **до** платёжного блока.
2. Стороне с `country != RU/РФ` росс. ИНН не подставлять (просьба SLAI).
- Файл: `pipeline/normalize/inn-recovery.ts`.
- Компромисс: «банк» в названии стороны («ООО Банк-Сервис») схлопнет окно → недобор ИНН. Безопасно:
  функция — best-effort fallback на флапы LLM, а промах в другую сторону даёт ложно-зелёную сверку.

### FIX-C (P1) — единое написание типа (**только после FIX-A**)
Дотянуть `normalizeSlugForApi` на **все** outbound-пути, включая `documents[]` (сегменты) и
`extracted`-payload. Сейчас SLAI видит оба написания → нормализатор покрывает не все выходы.
- Файлы: `src/types/slug-normalize.ts` (точка есть), места сборки `documents[]` + webhook payload.
- **Гейт приёмки:** сначала подтвердить, что маршрут появился у строчной ветки (FIX-A), иначе
  правка просто спрячет расхождение.

### ~~FIX-D (P1) — `file_name` уходит в LLM~~ → **НЕ ПОДТВЕРДИЛОСЬ, правка не нужна**

> ⚠️ **Исправление первой редакции этого документа (2026-07-17).** Здесь было заявлено, что имя файла
> уходит в LLM с extract-запросом со ссылкой на `llm/http-client.ts:126`. **Это неверно** — я прочитал
> строку, не проверив, в каком она методе.

Проверено по коду:
- `http-client.ts:126` (`file_name`) находится внутри **`classifyWithCatalog()`** → эндпоинт
  **`/v1/classify`**, а не extract. Для классификации имя файла — легитимный взвешенный сигнал
  (`classifier/types.ts:39`), и оно там и должно быть.
- `extract()` (`http-client.ts:134`) принимает `{text, schema, hint, promptOverride, includeDebug,
  imagePath}` — **`fileName` в сигнатуре нет**. `llmExtract()` (`parsers/llm-extractor.ts:18`) тоже:
  `(llm, rawText, schema, hint, expectedFields, …)`.
- `fileName` объявлен ТОЛЬКО на `LlmCatalogClassifyInput` (`llm/types.ts:13-16`).

**Вывод: канала «имя файла → extracted» не существует. Утечки нет.**

**Настоящее объяснение улики SLAI:** БКТ **именует файлы по машине** —
`956MRD_441YNP_205C_SUMEIRE.pdf` = `<тягач>_<прицеп>_<заказ>_<грузоотправитель>`. Машина совпала с
именем файла потому, что **имя произведено от машины**, а не машина от имени. Причинность обратная.

**Что сказать SLAI честно:** извлечению имя файла недоступно; на классификацию (выбор типа) оно влияет
осознанно и ограниченно. Если у них есть корпус с намеренно «мусорными» именами — прогон подтвердит,
но менять в извлечении нечего.

### FIX-E (P2) — контаминация блока перевозчика
**Улика:** заказ #5, `CMR.jpeg` → `carrier.phone = "balterex@mail.ru"` (почта **отправителя**
SIA BALTEREX). В один блок слиты поля разных лиц.
- `phone` не должен принимать e-mail (тип поля), и поля одной стороны не переносить в блок другой.
- По `carrier.inn` (= ИНН покупателя) претензий нет: при `FCA Тересполь` покупатель забирает груз
  своим транспортом — вероятный бизнес-факт. Проверить нечем (у покупателя в счёте только ИНН без
  имени) — **не «чинить» вслепую**.

## Ответы на вопросы SLAI

1. **«Маршрут реально отсутствует в бумаге или не извлекается?»** — **Есть в бумаге, не извлекается.**
   Причина найдена (FIX-A): строчной ветке никогда не давали CMR-схему, поля маршрута просто не
   запрашивались у модели. Те же физические бланки по верхней ветке отдают маршрут корректно
   (3/3). Ждать маршрут можно — после FIX-A.
2. **«`_field_confidence` заполняется?»** — **Да.** `orchestrator.ts:624` → `processFieldConfidence()`
   на каждом сегменте, возвращает `fieldConfidence` рядом с `extracted`. Гейтить по нему ИНН/стороны
   можно. Точное имя поля в контракте и покрытие по веткам — подтвердить перед тем как SLAI на него
   завяжется.
3. **«Схема 1.4 `_deep`: планируется ли выравнивание набора полей между ветками одного типа?»** —
   Это ровно FIX-A: после него обе ветки идут через один типизированный парсер и одну схему, т.е.
   набор полей выравнивается **по построению**, а не отдельной синхронизацией. `_deep` (второй ярус)
   к этому отношения не имеет — он про нераспознанный остаток.

## Приёмка

- **FIX-A:** у всех 11 CMR (обе ветки) есть `loading_place`/`delivery_place`/`border_crossing`;
  `number` = номер бланка, не «CMR» и не имя перевозчика. Маршрут строится у **6 заказов из 6** (сейчас 3/6).
- **FIX-B:** `inv_1.jpeg`/`pac_1.jpeg` → `seller.inn ≠ 7707083893`; у иностранного продавца росс. ИНН нет.
- **FIX-C:** во всех выходах ровно одно написание `cmr` (**после** проверки FIX-A).
- ~~FIX-D~~ — снят, канала «имя файла → extracted» не существует (см. выше).
- Регресс: `invoice`-ветка (не задета FIX-A) и ltk AUMA композиты 18/18 — не должны просесть.
