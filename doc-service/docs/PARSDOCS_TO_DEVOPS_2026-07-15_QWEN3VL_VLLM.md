# ТЗ DevOps: перенести vision-модель `qwen3-vl:32b` с Ollama на vLLM

**Дата:** 2026-07-15
**От:** parsdocs (dev)
**Кому:** DB Support / DevOps (владелец GPU-бокса `10.10.33.10`)
**Приоритет:** высокий — это потолок производительности разбора (см. §1).

---

## 1. Проблема (замерено на проде 2026-07-15)

Разбор упирается в **~30 документов/час**. Диагностика по живым задачам показала:
время ест **не OCR, а LLM-извлечение по картинке**:

| Этап | Время |
|---|---|
| OCR (tesseract) | ~31 с — норма |
| **LLM-извлечение (`qwen3-vl:32b`)** | **434–507 с (7–8 мин/док)** |

Причина: для сканов включён hybrid-routing → извлечение идёт **по изображению**
через тяжёлую 32B vision-модель, а она обслуживается **Ollama** (`10.10.33.10:11434`),
которая **сериализует** запросы (один за другим). 7–8 мин × 4 параллельных слота =
**~30 док/час**. Это ровно тот потолок, на который жалуется бизнес.

Для контраста: **текстовая** модель `qwen3.6` уже переведена на **vLLM**
(`10.10.33.10:8100`) и работает быстро. Осталось сделать то же для vision.

## 2. Что просим сделать

**Поднять `Qwen3-VL 32B` на vLLM** — по образцу уже работающего `qwen36-vllm`
(тот же приём: vLLM как OpenAI-совместимый сервер, отдельный порт). Модель —
**та же самая** (`qwen3-vl:32b` / `Qwen/Qwen3-VL-32B-Instruct`), меняем только
движок обслуживания Ollama → vLLM. Точность разбора не меняется, меняется скорость.

vLLM даёт continuous batching + параллельный KV-cache → параллельные запросы
обрабатываются вместе, а не по очереди. Ожидаемый выигрыш на этом этапе — **кратный**
(текстовую модель тот же переход ускорил ~13× под нагрузкой).

## 3. Технические требования

- **Модель:** Qwen3-VL 32B Instruct (мультимодальная, vision). Квантизацию (FP8 / AWQ /
  GPTQ) — на усмотрение DevOps под VRAM-бюджет; главное, чтобы это был тот же
  Qwen3-VL 32B, что сейчас в Ollama (сохранить качество).
- **Движок:** vLLM (та же версия, что для `qwen36-vllm`), OpenAI-совместимый endpoint
  `/v1/chat/completions` с поддержкой **image_url** (data-URL, JPEG) — parsdocs шлёт
  картинку страницы как base64 data-URL.
- **Порт:** отдельный от `qwen36-vllm` (тот на `8100`) — например **`8101`**.
  Итог: `http://10.10.33.10:8101/v1`.
- **VRAM / сосуществование:** бокс Blackwell 96 GB уже держит `qwen36-vllm`. Нужно
  решить, влезают ли обе модели одновременно (share GPU через `--gpu-memory-utilization`)
  или нужна вторая карта / другой сплит. Это ваша зона — мы не знаем текущий бюджет
  памяти. Если обе не влезают на одной карте — подскажите, обсудим приоритеты.
- **Мультимодал-лимиты vLLM:** `--limit-mm-per-prompt image=1` (нам достаточно 1
  картинки на запрос), разумный `--max-model-len` (документы у нас усечены до ~12k
  символов текста + 1 картинка; хватит контекста как у текущей Ollama-конфигурации).
- **systemd-сервис** (как `qwen36-vllm`), автозапуск, health-check.

## 4. Контракт вызова (что шлёт parsdocs)

parsdocs ходит на модель **не напрямую**, а через свой `inference-service`
(OpenAI-совместимый прокси). Ему нужен только рабочий `/v1/chat/completions`,
принимающий сообщения вида:

```
messages: [{role:"user", content:[
  {type:"text", text:"<промпт OCR/extract>"},
  {type:"image_url", image_url:{url:"data:image/jpeg;base64,..."}}
]}]
```

Ответ — обычный OpenAI chat completion. Никаких особых заголовков/форматов.
(На стороне parsdocs уже включено сжатие картинки перед отправкой — vLLM получит
уменьшенное изображение, это дополнительно снизит нагрузку.)

## 5. Что сделает parsdocs-сторона ПОСЛЕ (быстро, наша зона)

Как только vLLM-endpoint `10.10.33.10:8101/v1` поднят и отвечает — с нашей стороны
это **минутная правка** (по образцу строки `qwen36-vllm`): в `provider_settings`
строке `local-qwen3-vl-32b` проставить
`extra.backend='openai_compat'` + `extra.upstream_base_url='http://10.10.33.10:8101/v1'`.
TTL-кэш провайдера подхватит за минуты (или мелкий рестарт api/worker). Никакого
редеплоя кода не требуется.

**Дайте знать endpoint (IP:порт) — и мы переключаем.**

## 6. Приёмка

- Одиночный vision-вызов (OCR страницы / extract по картинке) возвращается за
  **секунды–десятки секунд**, не минуты.
- Параллельные запросы обрабатываются батчами (не сериализуются как в Ollama).
- Разбор скана «под ключ» уходит с ~8 мин к целевым ~1–2 мин; пропускная
  способность — с ~30/час в разы выше.
- Точность извлечения — не хуже текущей (модель та же).

## 7. Откат

Ollama-обслуживание `qwen3-vl:32b` (`10.10.33.10:11434`) **оставить как есть** —
это наш fallback. Если с vLLM что-то не так, parsdocs-сторона возвращает
`upstream_base_url` на Ollama одной правкой строки. Ничего не ломаем необратимо.

---

**Резюме:** нужен `Qwen3-VL 32B` на vLLM (порт напр. `8101`, OpenAI-совместимый,
multimodal), по образцу существующего `qwen36-vllm:8100`. Это снимает главный
тормоз разбора. Дальнейшее переключение — за минуту на нашей стороне.
