## 2. P1-A: Word-таблицы → Markdown (качество извлечения)

**Проблема.** `mammoth.extractRawText` разворачивает таблицы в поток строк:
ячейки склеиваются через `\n`, колонки исчезают. LLM видит «список слов» вместо
«строка × колонка». В ВЭД-доках таблица — это и есть полезная нагрузка
(items, характеристики, реквизиты). Прямое следствие — 4-полевые SDS из батча.

**Решение.** Перейти с `extractRawText` на `mammoth.convertToHtml` + лёгкий
HTML→Markdown конвертер (только `<table>/<tr>/<td>` → pipe-таблицы, `<p>` → абзацы,
остальные теги — strip). Никаких новых тяжёлых зависимостей: свой конвертер
~60 строк или `turndown` (лёгкий, без DOM).

Выход движка становится:
```
Абзац текста...

| Показатель | Значение | Ед. |
|---|---|---|
| ... | ... | ... |
```

**AC:**
1. docx с таблицей N×M отдаёт в OCR-текст все N строк с сохранёнными колонками.
2. Регресс-прогон 4 SDS-доков батча 232: fill ≥ прежнего, у «4-полевых» — рост.
3. Обычные (бестабличные) docx — текст эквивалентен текущему (без регресса).
4. Классификатор не деградирует (маркеры типов не исчезают из текста).

**Оценка:** 0.5–1 день.

## 3. P1-B: «Скан в ворде» → vision-fallback (полнота приёма)

**Проблема.** Типовой паттерн российского ЭДО: в .docx вставлена картинка-скан,
текста почти нет. Сейчас два плохих исхода:
- текст пустой → confidence 0 → дальше по цепочке движков, но **ни tesseract, ни
  vision-llm не принимают docx-mime** → `all OCR engines failed`;
- текста чуть-чуть (шапка) → confidence 1.0 → движок «принят» → LLM извлекает
  из огрызка → почти пустой результат, а мы даже не узнали.

**Решение** (два шага в одном пайплайне):
1. **Извлечение медиа:** docx/xlsx — это zip: вытащить `word/media/*`,
   `xl/media/*` (для .doc — catdoc не умеет, см. примечание). Картинки ≥ N KB
   (фильтр логотипов/печатей, порог ~50 KB — уточнить на корпусе) считать
   «страницами».
2. **Гибридный текст:** если суммарный извлечённый текст < MIN_TEXT_CHARS
   (порог ~200, уточнить) **и** есть крупные картинки → прогнать картинки через
   существующий vision-llm движок (qwen3-vl:32b) и склеить: `[текст docx] +
   [vision-текст картинок]`. Если текста достаточно — картинки не трогаем
   (не жечь GPU зря).

Примечание: для legacy .doc извлечение картинок дорого (OLE-парсинг) — в v2
скоуп только docx/xlsx (zip-контейнеры). .doc с картинками остаётся as-is.

**AC:**
1. docx-обёртка вокруг скана (текст < 200 chars, картинка > 50 KB) → результат
   содержит vision-текст, тип классифицирован, extraction непустой.
2. Обычный текстовый docx → vision НЕ вызывается (по логу `ocr_engine` без vision).
3. Порог/включение — env-флаги (`OFFICE_IMAGE_FALLBACK_ENABLED`, дефолт on,
   `OFFICE_IMAGE_MIN_KB`, `OFFICE_MIN_TEXT_CHARS`).
4. Живой тест: `ИВТТ-000041/Документы для таможни/Тех.описание...docx` (829 KB —
  почти наверняка картиночный) проходит с непустым извлечением.

**Оценка:** 1–2 дня (медиа-извлечение просто, аккуратность — в склейке и порогах).

## 4. P2: три доводки

### 4.1 Похоронить 2 исторических .doc-трупа
Два `x-cfb` со статусом `failed` («all OCR engines failed») — упали ДО ввода
catdoc. Действие: re-ingest обеих строк (переобработка существующего файла).
Если catdoc/antiword снова не осилят — файл, вероятно, не Word-CFB, а что-то
экзотическое: тогда фиксируем понятную ошибку пользователю и **не** тащим
LibreOffice в образ ради двух файлов (решение: держать лёгкий стек).
**AC:** обе строки в терминальном статусе с честным исходом (done или явная
ошибка формата). **Оценка:** 0.5 дня.

### 4.2 Колонтитулы docx
Реквизиты (ИНН/КПП/банк) в русских договорах часто живут в футере — mammoth
их не отдаёт. При переходе на convertToHtml — проверить; если не отдаёт,
дочитать `word/header*.xml`, `word/footer*.xml` напрямую (это простой XML)
и приклеить секцией `=== Колонтитулы ===`.
**AC:** договор с ИНН в футере отдаёт ИНН в OCR-текст. **Оценка:** 0.5 дня.

### 4.3 Multi-sheet: живой прогон F5
Механика есть (per-sheet pages → `tryMultiDoc`, orchestrator:446), но на живом
многолистовом Excel из ВЭД её не проверяли. Тест-кейс лежит в 262_00040:
`ZL251218-1 发票 箱单.xls` — «инвойс + упаковочный лист» одним файлом (发票 =
invoice, 箱单 = packing list). Прогнать, посмотреть, что вышло: 2 документа
(CI + PL) или один тип «по первому листу».
**AC:** файл даёт либо корректный multi-doc (2 типа), либо заведённый баг с
диагнозом. **Оценка:** 0.5 дня (это проверка, не разработка).

## 8. Порядок работ (предлагаемый)

1. **P1-A** Word-таблицы → Markdown (0.5–1 д) — самый дешёвый прирост качества.
2. **P1-B** vision-fallback для картиночных docx/xlsx (1–2 д).
3. **P2** три доводки (суммарно ~1.5 д).
4. **P3** — фоном, по одному пункту в свободные окна.

Суммарно ядро (P1+P2): **~4 рабочих дня**. Каждый пункт деплоится отдельно,
за флагом где уместно, с регресс-прогоном на корпусе §7.
