## 0. Контекст: кто пользователь и какие роли

UI v2 — SPA `doc-service/ui` (React + TS + Tailwind + React Query +
react-pdf + Vite). Маршруты — `ui/src/App.tsx`.

**Персона «оператор»** (из UX-аудита): не разработчик; его работа —
разбирать очередь `needs_review`, глазами сверять извлечённые поля с
изображением документа и либо править, либо одобрять. Скорость и
отсутствие «ловушек» (случайно потерять правки, случайно нажать
необратимое) — важнее украшений.

**Модель ролей** (`src/authz.ts`, frontend читает `/users/me.role`):

| Роль | Право | В UI |
|---|---|---|
| `super_admin` | всё, кросс-орг (root API-key) | все разделы |
| `org_admin` (`admin`) | запись в своей орг. | все разделы |
| `manager` | запись в назначенных проектах | оператор: работа с очередью |
| `viewer` | только чтение | оператор: смотрит, не пишет |

Признак оператора во фронте: `App.tsx:103` →
`isOperator = role === 'manager' || role === 'viewer'`. **`viewer`
дополнительно не должен видеть кнопки записи** (одобрить / править /
загрузить / reprocess) — см. F9.

## 1. F2 — структурный пофайловый редактор полей (наибольший)

**Проблема (UX-аудит F2):** редактирование `extracted` — это голый
`<textarea>` с сырым JSON. Оператор обязан понимать синтаксис JSON,
рискует сломать структуру, не видит какое поле «слабое» (low-confidence),
не понимает какие поля вообще ожидаются для типа.

**Решение — schema-driven редактор:**

1. **Все 30 типов** (решение принято): форма строится из схемы полей типа
   документа, а не хардкодом. Источник схемы — реестр
   [`DOCUMENT_TYPES.md`](DOCUMENT_TYPES.md); на бэке поля типа отдаёт
   `GET /document-types/:type` (поля, label, tier, validator). Фронт
   рендерит поле под его тип значения (строка / число / дата / массив /
   вложенный объект — напр. позиции инвойса, `total`-блок).
2. **Подсветка уверенности по полю:** для каждого поля показываем
   `_field_confidence[<path>]` цветом/бейджем (зелёный ≥0.85, жёлтый
   0.60–0.85, красный <0.60). Пороги — из единого источника (см. «мелочь
   /  confidence thresholds»), не хардкодить в трёх местах.
3. **Правка поля = «проверено человеком».** При ручном изменении значения
   поля его `_field_confidence` выставляется в **1.0** (решение принято:
   «проверено человеком»). Незатронутые поля сохраняют свою уверенность.
4. **«Сохранить ≠ одобрить» — две кнопки** (решение принято):
   - **Сохранить** — записать правки, **оставив** job в `needs_review`
     (оператор поправил часть, хочет вернуться позже / отдать коллеге).
   - **Одобрить** — `needs_review → done` (можно с несохранёнными
     правками: «сохранить и одобрить» одним действием).
   - ⚠️ **Требуется доработка бэка** (см. §8.1): сейчас
     `PATCH /jobs/:id/extracted` (`src/routes/jobs.ts:742`,
     `applyExtractedCorrection` в `src/storage/jobs.ts:365` —
     `status = 'done'` **всегда**) принудительно переводит в `done`.
     Режима «сохранить, не выходя из `needs_review`» сейчас нет.
5. **Незнакомый тип → форма по распознанным полям** (решение принято):
   если тип вне реестра 30, рендерим форму из фактически присланных полей
   `extracted` (а **не** сырой JSON). Переключатель «структурно ↔ JSON»
   остаётся только как отладочный fallback. Dirty-guard (F3) — в обоих
   режимах.

**Критерии приёмки F2:**
- [ ] Для любого из 30 типов открывается форма с ожидаемыми полями
  (не только теми, что пришли в `extracted`), пустые ожидаемые поля видны.
- [ ] Каждое поле показывает свою уверенность цветом; пороги едины с
  остальным UI.
- [ ] Правка поля → его `_field_confidence` = 1.0 в payload PATCH.
- [ ] Кнопка «Сохранить» оставляет статус `needs_review`; «Одобрить»
  переводит в `done`. (зависит от §8.1)
- [ ] Вложенные структуры (позиции, total) редактируются без сырого JSON.
- [ ] Документ незнакомого типа открывается формой по распознанным полям
  (сырой JSON — только в отладочном переключателе).
- [ ] Dirty-guard предупреждает при уходе с несохранёнными правками
  (переиспользовать F3).
- [ ] `viewer` видит значения, но поля недоступны для ввода, кнопок
  записи нет.

## 2. F5 — клавиатурная разборка очереди + навигация по PDF

**Проблема (F5):** очередь разбирается только мышью; на потоке это
медленно. У PDF нет клавиатурной навигации.

**Решение — горячие клавиши** (working-default, финал — см. «открытые
вопросы»):

| Клавиша | Действие | Где |
|---|---|---|
| `j` / `↓` | следующий документ | Review / JobDetail |
| `k` / `↑` | предыдущий документ | Review / JobDetail |
| `Enter` | открыть выделенный | Review |
| `a` | одобрить текущий | Review / JobDetail |
| `e` | в режим правки | JobDetail |
| `[` / `]` | пред./след. страница PDF | JobDetail |
| `+` / `-` | зум PDF | JobDetail |
| `Esc` | закрыть / выйти из правки | везде |
| `?` | шпаргалка по клавишам | везде |

**Критерии приёмки F5:**
- [ ] В Review `j/k` двигают выделение, `Enter` открывает, `a` одобряет.
- [ ] Клавиши не срабатывают, когда фокус в поле ввода/textarea.
- [ ] `?` показывает список клавиш.
- [ ] `viewer` — навигационные клавиши работают, мутирующие (`a`,`e`) нет.

## 3. F6 — единый контракт массовых операций

**Проблема (F6):** массовое одобрение в Review падает на первой ошибке
(один сбойный job рвёт всю пачку), поведение неконсистентно между
страницами.

**Решение:** все bulk-операции (approve, reprocess, redeliver) — через
`Promise.allSettled`-семантику: выполнять до конца, собрать результат
`{ успешно: N, ошибки: [{id, причина}] }`, показать сводку, **не**
откатывать успешные. Список ошибок — с возможностью повторить только
неуспешные.

**Критерии приёмки F6:**
- [ ] Bulk approve 10 шт., где 1 в неподходящем статусе → 9 одобрены,
  1 в списке ошибок с причиной; пачка не прервана.
- [ ] Сводка показывается единообразно во всех bulk-точках.
- [ ] «Повторить неуспешные» работает по оставшимся.

## 4. F8 — навигация пред./след. документ на детальной

**Проблема (F8):** из JobDetail нельзя перейти к соседнему документу той
же выборки — приходится возвращаться в список.

**Решение:** на JobDetail кнопки/клавиши «← пред. | след. →» в пределах
контекста, из которого пришёл (фильтр Jobs или очередь Review),
с сохранением фильтров/сортировки. Контекст передавать через состояние
навигации или query-параметр.

**Критерии приёмки F8:**
- [ ] Из Review переход по соседям не выходит за пределы очереди.
- [ ] Из отфильтрованного Jobs соседи — в пределах фильтра.
- [ ] На границах списка кнопка дизейблится (не циклится молча).
- [ ] Совместимо с горячими клавишами F5 (`j/k`).

## 5. F9 — ролевые гейты (закрыть админ-разделы)

**Проблема (F9):** админские разделы доступны по прямому URL даже
оператору — роутер не проверяет роль, только наличие токена
(`App.tsx:79 RequireAuth` смотрит лишь `isAuthenticated()`). Бэк права
проверяет, но UI показывает оператору то, что он не может использовать,
и пугающие 403.

**Решение — гейт на роль в роутере + скрытие пунктов меню.**
Working-default разметки (финал — «открытые вопросы»):

| Маршрут | Минимум роли |
|---|---|
| `/` лендинг, `/review`, `/jobs`, `/jobs/:id` | любой авторизованный |
| `/audit-log`, `/test-lab` | любой авторизованный (**решение владельца**: лабораторию и журнал видят и операторы) |
| `/upload` | `manager`+ (write) |
| `/document-types`, `/providers`, `/reference-lists`, `/tenants`, `/settings` | `admin`+ (разделы конфигурации) |

Поведение: пункт меню скрыт, если роль ниже; прямой переход по URL →
редирект на безопасный лендинг (или экран «нет доступа»), а не белый
экран/403. `viewer` везде в read-only: write-кнопки скрыты (в т.ч. кнопка
«прогнать» в Тест-лаборатории — смотреть может, запускать прогон — с
`manager`+).

**Критерии приёмки F9:**
- [ ] `viewer`/`manager` не видят **разделы конфигурации** в меню (типы,
  провайдеры, справочники, организации, настройки).
- [ ] `viewer`/`manager` **видят** Тест-лабораторию и Журнал действий.
- [ ] Прямой URL `/tenants` оператором → редирект, без 403-страницы.
- [ ] `viewer` не видит кнопок «одобрить/править/загрузить/reprocess» и не
  может запустить прогон в Тест-лаборатории.
- [ ] `admin`/`super_admin` видят всё как раньше.

## 6. F10 — внутримодальные подтверждения вместо native confirm/alert

**Проблема (F10):** действия с внешними последствиями
(reprocess, redeliver-webhook, удаление) используют браузерные
`confirm()/alert()` — некрасиво, легко прокликать, не консистентно.
Пересекается с правилом CLAUDE.md «действия с внешними последствиями —
через явное подтверждение».

**Решение:** единый компонент-модал подтверждения (заголовок, описание
последствия, имя объекта, кнопки «Отмена»/«Подтвердить», у необратимых —
акцентный destructive-стиль). Заменить все `window.confirm/alert`.

Действия, требующие модала (по бэку подтверждено):
- `POST /jobs/:id/reprocess` (`jobs.ts:924`) — переобработать.
- `POST /jobs/:id/redeliver-webhook` (`jobs.ts:844`, `?force=true`) —
  повторно отправить вебхук в SLAI (внешний эффект! — особо явно).
- Удаление сущностей в админ-CRUD.

**Критерии приёмки F10:**
- [ ] Нет ни одного `window.confirm/alert` в потоках мутаций.
- [ ] Redeliver-webhook отдельно предупреждает «повторно отправит данные
  во внешнюю систему».
- [ ] Модал закрывается по `Esc`/клику вне/«Отмена»; «Подтвердить»
  дизейблится на время запроса.

## 8. Зависимости от бэка

### 8.1 Режим «сохранить без одобрения» (блокирует F2)

Сейчас единственный путь записи правок — `PATCH /jobs/:id/extracted`,
который **всегда** ставит `status='done'`
(`src/storage/jobs.ts:365-372`). Для «Сохранить ≠ одобрить» нужен режим,
сохраняющий правки **без** выхода из `needs_review`.

Варианты (выбрать на ревью — см. «открытые вопросы»):
- **(A)** Параметр у существующего PATCH:
  `PATCH /jobs/:id/extracted?keep_status=true` — пишет `extracted` +
  `extracted_corrected_at`, статус не трогает; повторная доменная
  валидация выполняется как сейчас.
- **(B)** Отдельный эндпоинт «черновик правок» + существующий PATCH как
  «сохранить и одобрить».

Рекомендация: **(A)** — минимальная и обратносовместимая (без параметра —
старое поведение).

### 8.2 Схема полей типа для F2

F2 опирается на `GET /document-types/:type` (поля/label/tier/validator).
Убедиться, что эндпоинт отдаёт полный список ожидаемых полей и тип
значения каждого (для выбора виджета ввода). Если чего-то нет — мелкая
доработка бэка.

## 9. Мелочь (polish, низкий приоритет)

- **Upload (низкий приоритет — UI-загрузка это заглушка):** на практике
  документы пойдут через внешние программные продукты по API; ручная
  загрузка в UI — вспомогательная. Поэтому per-file опции **не делаем** —
  одна настройка на всю пачку (`redact_pii`-галка, опц. тип-подсказка;
  F4 redact уже в metadata). Из обязательного — убрать хрупкий
  `setTimeout`-автопереход (переход по факту ответа, не по таймеру).
- **Свёрнутый сайдбар** теряет число-счётчик очереди — показывать бейдж и
  в свёрнутом виде.
- **Пороги confidence** дублируются в 3 местах — вынести в один модуль-
  константу/конфиг, использовать в Review, F2-редакторе, бейджах.
- **ValidationBanner:** клик по проблеме → скролл/фокус на проблемное поле.
- **PdfViewer:** нет поиска/поворота/клавиш; зум сбрасывается при ресайзе —
  сохранять масштаб; добавить поворот (для криво сканированных).
- **TierBadge:** опечатка «valиdators» (смешанная кириллица/латиница).
- **JobDetail:** нет скелетон-лоадера; уверенность в шапке — текстом, а не
  баром (привести к виду как у полей).

## 10. Решения по открытым вопросам (закрыто 2026-06-02)

1. **F2 / режим сохранения** → вариант **(A)**:
   `PATCH /jobs/:id/extracted?keep_status=true` пишет правки, не трогая
   статус; без параметра — прежнее поведение («сохранить и одобрить»).
   _(техническое решение)_
2. **F9 / доступ к служебным разделам** → **Тест-лаборатория и Журнал
   действий видны и операторам** (`manager`/`viewer`); экраны конфигурации
   (типы, провайдеры, справочники, организации, настройки) — только админам.
   _(решение владельца)_
3. **F5 / раскладка клавиш** → утверждён набор §2 по умолчанию
   (`a`=одобрить, `e`=правка, `j/k`=навигация, `?`=шпаргалка); при
   необходимости переназначим позже. _(техническое решение)_
4. **F2 / незнакомый тип** → **форма по фактически распознанным полям**;
   сырой JSON — только отладочный fallback. _(решение владельца)_
5. **Upload / per-file vs batch** → **одна настройка на пачку**. Ручная
   загрузка в UI — заглушка; основной приём документов пойдёт через внешние
   продукты по API, поэтому в per-file опции не вкладываемся.
   _(решение владельца)_
6. **F10 / удаляемые сущности** → удаление через UI (с модалом-
   подтверждением) для админских CRUD-справочников (типы документов,
   провайдеры, записи справочников); организации/проекты/пользователи —
   с повышенным подтверждением. Финальный список уточняем при реализации
   админки. _(техническое решение)_

## 11. Порядок реализации (предложение)

1. **F9** (ролевые гейты) — безопасность, быстро, разблокирует корректный
   показ кнопок для остальных пунктов.
2. **§8.1** (бэк: save-без-одобрения) — разблокирует F2.
3. **F2** (редактор полей) — наибольшая ценность для оператора.
4. **F10** (модалы подтверждения) + **остаток F11** — внешние эффекты.
5. **F6** (единый bulk-контракт).
6. **F5** (клавиши) + **F8** (навигация по соседям) — скорость потока.
7. **Мелочь** — по остаточному принципу.
