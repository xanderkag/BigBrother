## 1. Назначение

Разбор **фотографий удостоверяющих документов** (паспорт РФ, водительское удостоверение, СТС, ПТС и др.) через **Vanga** (мультимодальная модель, on-prem) → структурированные поля → в карточки SLAI (**водитель / транспортное средство / контрагент**). Данные не покидают контур.

Отличие от разбора обычных документов (инвойсы/накладные): у удостоверений **фиксированный шаблон**, часть полей критичны (номер, ФИО), и это **персональные данные** (см. §7).

> **Vanga = parsdocs** (внутреннее имя). Поэтому фича **не строится с нуля** — едет на уже готовом пайплайне (vision-движок `qwen3-vl:32b`, LLM-классификатор, schema-extract, webhook, draft-паттерн, `redact_pii`). Что именно переиспользуется — §6.1. Реально новое — только ID-специфика (типы, MRZ-парсер, ПДн-хранилище).

## 2. Область — типы документов

| Тип | Код | Приоритет |
|---|---|---|
| Паспорт РФ (внутренний) | `passport_rf` | P0 |
| Водительское удостоверение РФ | `driver_license_rf` | P0 |
| СТС (свидетельство о регистрации ТС) | `vehicle_reg_sts` | P1 |
| ПТС (паспорт транспортного средства) | `vehicle_passport_pts` | P1 |
| Заграничный паспорт РФ | `passport_intl_rf` | P2 (есть MRZ) |

Прочие (СНИЛС, ИНН-документ) — по мере необходимости, тем же контрактом.

---

## 3. Контракт извлечения — поля по типам

Каждое поле — `{ value, confidence }`. Пустое/нечитаемое → поле отсутствует или `value: null`.

**`passport_rf` (паспорт РФ):** `surname`, `name`, `patronymic`, `sex`, `birth_date`, `birth_place`, `series` (4 цифры), `number` (6 цифр), `issued_by`, `issue_date`, `division_code` (XXX-XXX). ⚠️ У внутреннего паспорта РФ **MRZ нет** — извлечение только из печатных полей, точность ниже загранпаспорта.

**`driver_license_rf` (ВУ):** `surname`, `name`, `patronymic`, `birth_date`, `birth_place`, `number` (серия+номер), `issue_date` (4a), `expiry_date` (4b), `issued_by` (4c), `categories` (список: B, C, CE…), `special_marks`.

**`vehicle_reg_sts` (СТС):** `reg_number` (госномер), `vin`, `brand_model`, `year`, `vehicle_type`, `color`, `engine_number`, `chassis_number`, `body_number`, `sts_series_number`, `owner`.

**`vehicle_passport_pts` (ПТС):** `vin`, `brand_model`, `year`, `vehicle_type`, `engine_number`, `chassis_number`, `body_number`, `color`, `engine_power`, `pts_series_number`, `owners` (может быть несколько).

**`passport_intl_rf` (загран):** те же ФИО/даты + **`mrz`** (две строки, машиночитаемая зона с контрольными суммами — приоритетный источник).

## 4. Формат вывода (контракт ответа)

```json
{
  "document_type": "passport_rf | driver_license_rf | vehicle_reg_sts | vehicle_passport_pts | passport_intl_rf | unknown",
  "fields": { "<поле>": { "value": "...", "confidence": 0.0 } },
  "mrz": { "raw": ["<L1>", "<L2>"], "checksum_valid": true, "parsed": { } } ,
  "authenticity": { "suspect": false, "reasons": [] },
  "quality": { "blur": 0.0, "glare": 0.0, "crop_ok": true, "multiple_documents": false },
  "_issues": ["low_quality" | "partial_crop" | "multiple_documents" | "unreadable" | ...]
}
```

- `confidence` по каждому полю — обязателен (для «светофора» и порога ручной проверки).
- `mrz`/`authenticity` — где применимо (`null` если нет).
- `document_type: unknown` + `_issues` — если тип не распознан или фото нечитаемо (не падать). Совпадает с текущим `unknown`-контрактом parsdocs.

## 5. Требования к качеству

- **MRZ (загран) приоритетнее печатных полей** — там контрольные суммы.
- Пред-обработка фото (обрезка/выпрямление/контраст) — на стороне Vanga или отдельным шагом (уточнить §8).
- Порог: поля с `confidence` ниже порога тенанта → документ **на ручную проверку** (не автозаполнять «вслепую»). Переиспользуем существующие `needs_review`-пороги.
- Несколько документов в одном фото / обрезка / смаз → в `_issues`, на ручную.

### 5.1 Модель извлечения (решение 2026-07-06)

Паспорт = ПДн → **облачные VLM (GPT-4o / Claude vision) ИСКЛЮЧЕНЫ** (данные не покидают контур, 152-ФЗ). Только on-prem:

- **Дефолт-кандидат:** `qwen3-vl:32b` — тот же on-prem VLM, что уже работает в parsdocs (OCR-каскад, vision-fallback P1-B). RU-текст читает.
- **MRZ (загран) — НЕ через VLM:** машиночитаемая зона с контрольными суммами → **детерминированный парсер** (regex + checksum). VLM лишь считывает 2 строки; итоговые поля берём из checksum-валидированного разбора (надёжнее любой модели).
- **Внутренний паспорт РФ — MRZ нет** → чистый VLM по печатным полям, точность заведомо ниже → критичнее порог ручной проверки (§5).
- **Решение по модели — по замеру, не по догадке:** прогнать бенч на образцах (свои / обезличенные) — per-field точность, особенно `number` / ФИО / MRZ. Если `32b` не дотянет — на боксе (96 ГБ VRAM) поднять более крупный локальный VLM. Бенч-харнес у нас есть. **Без образцов решение не финализируем.**

## 6. Интеграция SLAI

Маппинг в реальные сущности (доп. поля — в `draftMeta` jsonb, как у существующего draft-паттерна):

| Тип | Сущность | Прямой маппинг | В `draftMeta` |
|---|---|---|---|
| `passport_rf` / `passport_intl_rf` | **Driver** | `fullName`(+last/name/middle), `passport` = «серия номер» | `passport{issued_by, issue_date, division_code, birth_date, birth_place, sex}` |
| `driver_license_rf` | **Driver** | `fullName`, `driverLicense` = `number` | `license{categories, issue_date, expiry_date, issued_by}` |
| `vehicle_reg_sts` / `vehicle_passport_pts` | **Car** | `carNumber` (СТС), `vin`, `brand`, `chassisNumber` | `sts/pts{series_number, owner, year, type, color, engine_number, body_number, power}` |

- **Draft-паттерн (переиспользуем):** результат создаёт/обновляет **черновик** Driver/Car (`isDraft=true`, `draftSource='vanga-id'`, `draftMeta`=полная выжимка), менеджер подтверждает — как с parsdocs-черновиками компаний/номенклатуры.
- **Привязка:** если фото загружено в контексте плеча/перевозчика — предзаполнить связь (водитель→плечо, ТС→перевозчик). Иначе — в общий пул черновиков «на разбор».
- **Доставка результата в SLAI:** тем же механизмом, что документы parsdocs (org-вебхук, schema 1.2), либо отдельный vision-эндпоинт — **уточнить у Vanga (§8)**.

### 6.1 Что переиспользуем (reuse-карта, подтверждено graphify 2026-07-06)

Vanga = parsdocs → ~70% механики уже есть, новое — только ID-специфика:

| Нужно по ТЗ | Готовое в parsdocs |
|---|---|
| VLM-извлечение on-prem | vision-движок `qwen3-vl:32b` (OCR-каскад, P1-B) |
| Классификация типа | LLM-классификатор + honest `unknown` |
| Поля `{value, confidence}` | schema-extract + `_field_confidence` |
| Порог ручной проверки | `needs_review`-пороги тенанта |
| Доставка в SLAI | org-вебхук + schema 1.2 |
| Черновик Driver/Car | draft-паттерн (`isDraft` + `draftMeta`) |
| Обезличивание ПДн | `redact_pii` + Yandex-PII-guard (§7) |

**Новое (нет как готовое):** ID-типы (§2), **MRZ-парсер** (§5.1), **ПДн-хранилище DOC-FILE-STORE-1** (§7, greenfield).

## 7. 🔴 Персональные данные (152-ФЗ) — обязательно

Паспорт/удостоверение = ПДн. До запуска согласовать с ответственным за ПДн:

- **Правовое основание / согласие** субъекта на обработку.
- **Хранение изображения — ХРАНИМ (решение владельца 2026-07-06), но защищённо:**
  - **зашифровано at-rest** — зашифрованный blob в S3-совместимом on-prem хранилище (MinIO, server-side encryption), НЕ в открытом volume;
  - **доступ по праву (RBAC)** — скан и паспортные поля отдельным правом, не всем подряд;
  - **срок хранения (TTL)** — lifecycle-правило на бакете, удаление по политике;
  - **аудит доступа** — кто и когда открывал скан удостоверения.
  - Реализация — **DOC-FILE-STORE-1** (⚠️ **greenfield**: спеки ещё нет, писать отдельной ТЗ; опирается на существующие `file_sha256` + слой хранения файлов parsdocs). Делаем общий file-store: для торговых доков + строже для удостоверений.
- **Маскирование в UI и логах** — не показывать/не логировать полный номер паспорта без права; в логах ПДн не писать.
- **Единая ПДн-политика, не третья.** Свести с уже существующими механизмами: `redact_pii` (флаг обезличивания на ingest) + PII-guard для внешнего OCR (`YANDEX_DISABLE_FOR_PII`, 152-ФЗ). НЕ плодить отдельную политику под удостоверения — **расширить общую**. (graphify 2026-07-06: эти три механизма сейчас в доках не сшиты — свести.)

## 9. Acceptance Criteria

- **AC1.** Vanga по каждому типу (§2 P0) отдаёт поля §3 с `confidence` в формате §4; нечитаемое → `unknown`+`_issues`, без падения.
- **AC2.** SLAI принимает результат и создаёт/обновляет **черновик** Driver/Car по маппингу §6; менеджер подтверждает.
- **AC3.** ПДн: скан удостоверения хранится **только** зашифрованно (blob, RBAC, TTL, аудит — §7); поля маскируются в UI/логах; доступ по праву. Незашифрованного или бессрочного хранения нет.
- **AC4.** Поля ниже порога уверенности → документ на ручную проверку (не автозаполнение).
- **AC5.** MRZ загранпаспорта разбирается детерминированным checksum-парсером; при невалидной контрольной сумме — `mrz.checksum_valid=false` + на ручную (§5.1).
