**Дата:** 2026-05-31

**Связано:** `SLAI_INTEGRATION_BACKLOG.md` (Слой 3 multi-instance был заморожен — теперь размораживаем), `CLAUDE.md` (CP7 multi-tenant CRUD готов в коде).

## TL;DR

Три эпика:

| ID | Что | Зачем | Размер |
|----|-----|------|--------|
| **MTI-1** | Multi-instance SLAI management — таблица всех инстансов с mass-actions + пресеты + copy+apply | сейчас каждый sandbox вручную: CLI script → token → envelope → PR в SLAI. На 3-5 инстансах это терпимо, на 20+ — не работает | M (2-3 дня UI + ½ дня backend) |
| **MTI-2** | Model preset bundles — один provider = pack моделей, per-job выбор + auto-routing | сейчас один provider = одна модель, "несколько моделей Anthropic" = 5 копий одной строки | M (1-2 дня backend + 1 день UI) |
| **MTI-3** | Unify key storage — один путь хранения LLM-ключа (UI Providers → БД → передаётся в inference per-request как body) | сейчас 3 места одновременно: provider_settings.api_key, ANTHROPIC_API_KEY в inference .env, X-LLM-Api-Key header. Пользователь вставляет ключ в UI, но parsdocs ходит в Anthropic с .env-ключом. Путаница | S (1-2 дня backend, миграция .env → fallback only) |

Все три — UX-улучшения, **не** меняют контракт с consumer'ами (SLAI). После реализации:
- Owner видит одну таблицу всех SLAI-инстансов, делает массовые операции
- Owner видит одну строку "Anthropic" с возможностью выбрать модель per-job/per-org/auto
- Ключ Anthropic живёт в **одном** месте (UI Providers), не дублируется

## MTI-1: Multi-instance SLAI management

### Сегодняшняя боль

Чтобы подключить новый SLAI-инстанс:

1. SSH к parsdocs хосту
2. `npm run provision:sandbox -- --name <slai-X>` через docker exec
3. Скопировать plaintext token из вывода
4. age encrypt под их public key
5. PR в их `SLAI_SECRETS_INBOX.md`
6. Дождаться когда они применят
7. Если нужно потом изменить webhook URL — повторить весь цикл

Не масштабируется. Нет ни одного экрана где владелец видит "все SLAI которые ко мне подключены".

### Требования

#### 1.1. Страница "Connected Consumers" (новая, либо расширение `/ui/tenants`)

Таблица с фильтром:

| ID | Имя | Тип клиента | Endpoint | Status | LLM mode | Last call | Token expires | Actions |
|----|-----|-------------|----------|--------|----------|-----------|---------------|---------|
| 9a3c… | slai-sandbox | sandbox | api.demo.sls24.ru | 🟢 active | BYO | 12 min ago | 90d | Edit / Rotate / Delete |
| ... | slai-prod-1 | production | api.sls24.ru | 🟢 active | shared anthropic | 2s ago | 60d | … |
| ... | slai-demo-2 | demo | demo2.sls24.ru | 🟡 idle 7d | shared anthropic | 7d ago | 30d | … |

**Источник данных:** объединение `organizations + organization_settings + personal_access_tokens + последний job из jobs (LATERAL JOIN)`.

#### 1.2. Mass actions (checkbox + bulk-bar)

- **Bulk enable BYO_LLM** — для выбранных org поставить `extra._byo_llm_required=true`
- **Bulk set webhook secret** — массово ротировать webhook_secret через S1 envelope от SLAI
- **Bulk rotate tokens** — пересоздать PAT для N юзеров одной транзакцией
- **Bulk delete** (с подтверждением) — снести testing-инстансы за раз

#### 1.3. Presets — шаблоны конфигурации

Один dropdown "Применить пресет":

- **sandbox** (default): `output=webhook`, retention 7d, rate-limit 60, `is_test=true`
- **production**: retention 30d, rate-limit 600, мониторинг alerts on
- **demo-readonly**: только classification, `mode=classify_only`, без extract

Кнопка "Create from preset" → форма с предзаполненными значениями. Можно сохранить **custom preset** под именем.

#### 1.4. Copy + apply

В строке инстанса — кнопка "Duplicate as new" → форма с настройками текущего, поменять только `name` и `webhook_url`. Token генерируется новый. Полезно для "вот ещё один такой же клиент".

#### 1.5. Per-org overrides (миграция нужна)

Сейчас `RATE_LIMIT_PER_MINUTE` и `FILE_RETENTION_DAYS` — глобальные ENV хоста. Добавить в `organization_settings`:
```sql
rate_limit_per_minute INT NULL,  -- NULL = inherit global
retention_days        INT NULL,  -- NULL = inherit global
```

И в `bearerAuthHook` / `file-cleanup-sweeper` респектить per-org значения с fallback на global.

#### 1.6. Secrets channel automation

Сейчас S2 envelope ручной (я генерирую токен → ты копируешь → шифруешь age → PR в SLAI repo).

Хочется кнопку "**Push token to SLAI inbox**" в UI:
- Owner один раз сохраняет SLAI's age public key в `connector_settings` (kind=`age_inbox`)
- При создании sandbox-тенанта parsdocs автоматически:
  - генерит token
  - шифрует под SLAI public key
  - POST'ит в их `xanderkag/SLAI` через GitHub API (создаёт PR в `SLAI_SECRETS_INBOX.md`)
- Owner видит "envelope sent ✅ wait for SLAI to apply"

Это убирает 3 ручных шага из 7-step cycle.

### Acceptance

- ✅ Таблица показывает 1+ org с реальными метриками (last_call, expires)
- ✅ Mass enable BYO для N org = одна транзакция (атомарно)
- ✅ Preset "production" применяется к новой org → правильные `organization_settings` + token
- ✅ Duplicate as new создаёт ОТДЕЛЬНУЮ org + ОТДЕЛЬНЫЙ token (не shared)
- ✅ Per-org rate-limit срабатывает в `bearerAuthHook` (тест: org-A с 60/min, org-B с 600/min → разные коды 429)
- ✅ Push to SLAI inbox создаёт PR в `xanderkag/SLAI` с envelope — owner видит link

### Не делаем (явно)

- Cross-org SSO для SLAI юзеров — каждый инстанс sandbox, не нужно
- Billing / usage tracking per-org — отдельная тема (Q будущего)
- Public marketplace tenants — это вообще другой продукт

## MTI-2: Model preset bundles

### Сегодняшняя боль

User в `/ui/providers` видит **`anthropic`** с моделью `claude-sonnet-4-5`. Хочет "ещё хочу claude-opus-4-7, claude-haiku-4-5". Сегодня единственный способ — **создать ещё 2 строки** в `provider_settings` с разными `id`:

```
anthropic              → claude-sonnet-4-5
anthropic-opus         → claude-opus-4-7
anthropic-haiku        → claude-haiku-4-5
```

Все 3 строки хранят **один и тот же API key** (3 копии envelope). Это противно: ключ дублируется, нет UI чтобы выбрать модель per-job красиво, при ротации ключа нужно править 3 строки.

### Требования

#### 2.1. Один provider = pack моделей

Изменить схему `provider_settings`:

```sql
ALTER TABLE provider_settings DROP COLUMN model;
ALTER TABLE provider_settings ADD COLUMN models JSONB NOT NULL DEFAULT '[]';
ALTER TABLE provider_settings ADD COLUMN default_model TEXT;
```

`models` — массив объектов:
```json
[
  { "name": "claude-sonnet-4-5", "alias": "sonnet", "vision": false, "cost_tier": "mid" },
  { "name": "claude-opus-4-7",   "alias": "opus",   "vision": true,  "cost_tier": "high" },
  { "name": "claude-haiku-4-5",  "alias": "haiku",  "vision": false, "cost_tier": "low" }
]
```

`default_model` — какую брать если в job не указано иное.

#### 2.2. Per-job model override

В `metadata._llm_model` (по аналогии с `_force_provider_id`):

```json
{ "metadata": { "_llm_model": "claude-opus-4-7" } }
```

Если задано — orchestrator подкладывает в body `model` поле inference-вызова, перебивая `default_model` провайдера.

Bonus: можно указать **alias** (`"opus"`) — резолвер найдёт в `models[]` по alias.

#### 2.3. UI Test Lab — dropdown моделей

Когда выбран provider `anthropic`, ниже появляется второй dropdown "Model":
- sonnet (claude-sonnet-4-5) **— default**
- opus   (claude-opus-4-7)
- haiku  (claude-haiku-4-5)

Custom — поле ввода для не-в-pack модели (вдруг новая вышла).

#### 2.4. Auto-routing по типу документа (опционально, после 2.1-2.3)

В `document_types` добавить:
```sql
ALTER TABLE document_types ADD COLUMN preferred_model TEXT;
```

Например:
- `customs_declaration` → `claude-opus-4-7` (сложный, нужна точность)
- `cash_receipt` → `claude-haiku-4-5` (простой, экономим)
- остальные → fall through на default_model провайдера

#### 2.5. Migration существующих данных

Migration 0033 (или next):
```sql
-- Сохранить существующее значение model как первый элемент массива models[]
UPDATE provider_settings SET
  models = jsonb_build_array(jsonb_build_object('name', model, 'alias', 'default')),
  default_model = model
WHERE model IS NOT NULL;
```

Backward compat для одной модели — gentle, существующие jobs не ломаются.

### Acceptance

- ✅ Один `anthropic` provider содержит pack из 3+ моделей
- ✅ В UI Test Lab dropdown показывает все 3 + custom-input
- ✅ `metadata._llm_model: "opus"` → реально вызывает opus (видно в `last_llm_call.model`)
- ✅ `preferred_model` для типа документа применяется автоматически
- ✅ Старые jobs (созданные до миграции) НЕ падают — backward compat работает

### Не делаем

- Multi-provider auto-fallback (если sonnet rate-limited → switch to opus автоматически) — отдельная тема (надёжность)
- Per-org разные `default_model` — следствие MTI-1 если понадобится

## MTI-3: Unify key storage

### Сегодняшняя боль

User в UI Providers вставил Anthropic ключ. Сохранил. Но Claude всё равно не зовётся, потому что:

- `inference-service/.env` = `BACKEND=stub` → Anthropic в принципе не вызывается
- Если переключить `BACKEND=claude` — inference возьмёт ключ **из `ANTHROPIC_API_KEY` в .env**, а **не из БД parsdocs**
- `provider_settings.api_key` (что user вставил в UI) → передаётся как `Authorization: Bearer` в inference (inter-service auth), не до Anthropic

То есть **3 разных места** для одного логически "Anthropic key":

| # | Где | Используется когда |
|---|-----|---------------------|
| 1 | `provider_settings.api_key` (БД parsdocs, AES-256-GCM, через UI Providers) | inter-service auth doc-service → inference (`Authorization: Bearer`) |
| 2 | `inference-service/.env::ANTHROPIC_API_KEY` (статика) | реальный вызов `api.anthropic.com` если `BACKEND=claude` |
| 3 | `X-LLM-Api-Key` header per-request (BYO LLM) | передаётся в body inference → реальный Anthropic call |

EXT-B (BYO LLM, путь #3) сделан правильно: ключ per-request в body, inference использует его. Но дефолт-провайдер (путь #1) ходит legacy-путём — через `Authorization` header, который к Anthropic вообще не имеет отношения.

### Требования

#### 3.1. HttpLlmClient передаёт api_key в body, а не в Authorization

`doc-service/src/pipeline/llm/http-client.ts`:

```typescript
// БЫЛО:
headers: { authorization: `Bearer ${this.opts.apiKey}` }

// СТАЛО:
body: { ...existing, llm_api_key: this.opts.apiKey, llm_provider: this.opts.provider }
```

То есть тот же путь что в EXT-B BYO LLM — `inline-credentials.ts` уже умеет.

#### 3.2. Inference читает body.llm_api_key для дефолт-вызовов

Backend `anthropic.py` / `openai_compat.py`:

```python
# БЫЛО:
if req.llm_api_key:                # только BYO
    use ephemeral_client(req.llm_api_key)
else:
    use self._client                # default из ANTHROPIC_API_KEY env

# СТАЛО:
key = req.llm_api_key or self._env_fallback_key
if not key: raise Error("no_key_configured")
use ephemeral_client(key)
```

`self._env_fallback_key` — `ANTHROPIC_API_KEY` из env, только если в body не пришло (для dev-окружения без UI Providers).

#### 3.3. Доступ к ключу через resolver

`provider-resolver.ts`: при выборе default-провайдера дёргает `provider_settings.api_key` (decrypt envelope) → передаёт в HttpLlmClient.opts.apiKey. Уже работает; нужно только убедиться что доходит до body, а не Authorization.

#### 3.4. UI Providers — tooltip про значение ключа

В форме edit Anthropic — текст под полем `API key`:

> Этот ключ передаётся в inference-service для вызова Anthropic API. Он
> приоритетнее статичного `ANTHROPIC_API_KEY` в `inference-service/.env`
> (который оставлен как fallback для dev-окружения). Шифруется AES-256-GCM
> envelope'ом перед записью в БД, расшифровывается только в момент вызова.

#### 3.5. Миграция

- Если у пользователя сейчас работает только через `ANTHROPIC_API_KEY` в .env, и `provider_settings.api_key` для anthropic пустой → ничего не меняется (fallback срабатывает).
- Если оба значения есть — приоритет за БД.
- Логирование при старте: `info "Anthropic key source: db / env_fallback / not_configured"`.

### Acceptance

- ✅ User вставил ключ в UI Providers → Anthropic вызовы реально идут с этим ключом (видно в metrics `extractor_llm_provider_calls_total{provider=anthropic, key_source=db}`)
- ✅ Если ключ в UI пустой, но в `ANTHROPIC_API_KEY` env — fallback работает (backward compat)
- ✅ Если ни там ни там — внятная ошибка `no_key_configured` (не silent failure)
- ✅ `Authorization: Bearer ...` между doc-service и inference больше не несёт API key (либо убран совсем, либо переиспользуется для своей задачи: inter-service auth с отдельным секретом)
- ✅ UI tooltip explain'ит куда уходит ключ

### Не делаем

- Multi-key одновременно (один Anthropic key для одних org, другой для других) — это MTI-1 (per-org) + MTI-3 (механизм). Связка отдельно.
- Bring-your-own model name validation — модель просто проксируется в Anthropic, если у них её нет — они вернут 400.

### Зависимость от других MTI

- **Независимо от MTI-1**: можно делать первым (1-2 дня), сразу убирает путаницу
- **Перед MTI-2**: чтобы pack-of-models имел смысл — должен работать единый ключ. MTI-3 → MTI-2.

### Что не блокирует пилот WW-23

- Пилот стартует с **ОДНИМ** инстансом SLAI (sandbox) — текущего CLI достаточно
- Anthropic с одной моделью (sonnet-4-5) — sandbox + Test Lab покрывают use case
- MTI-1 / MTI-2 пригодятся когда появится 2-й и 3-й инстанс SLAI или захочется expensive-on-demand модели

## Open questions

1. **GitHub API token для Push to SLAI inbox** — нужен PAT с write-доступом в `xanderkag/SLAI`. Кому хранить? `connector_settings.kind='github_app'`?
2. **Если SLAI ротирует свой age public key** — старые pending envelope становятся не-расшифровываемы. Нужен механизм "resend all pending under new key"?
3. **Auto-routing по типу документа vs cost cap** — что приоритетнее, если у клиента есть месячный бюджет? Это уже billing-вопрос, отложить.
4. **Custom presets shared между instances parsdocs** — или per-host? (если когда-то будет 2-й parsdocs хост — sync presets не нужен)
5. **Multi-instance в UI: "видеть от обоих parsdocs хостов" (Asha + kb-docker)** — нет, каждый хост видит только свои тенанты. Кросс-хост — это уже federation, отдельный проект.
