# Vanga как интеграционный хаб (control-plane для внешних API) — vision

> **Статус:** видение, развилки сняты owner'ом 2026-06-25. Точка сборки для роадмапа.
> Связано: `ROADMAP.md` (EXT-LLM-PROXY-C, MTI-1), `TECH_DEBT.md`.

## Идея
Vanga (parsdocs) — не только центр разбора документов, но и **единая точка подключения,
контроля, мониторинга и учёта внешних API-интеграций** для нескольких внутренних
потребителей. Ключи держим централизованно, потребители ходят через наш шлюз по токену,
каждый вызов метерим и **атрибутируем потребителю** (инстансу/проекту).

## Решённые развилки (owner, 2026-06-25)
1. **Кто клиент** — внутренний **мульти-потребитель**, модель = **cost-attribution / chargeback**,
   НЕ внешние продажи. Потребители: СЛАЙ (несколько продуктов, иногда несколько инстансов) +
   проекты других подразделений (пример: подразделение дёргает DaData по ИНН). Нужно видеть
   «кто сколько чего использовал» по `потребитель × коннектор`.
2. **Единица оплаты** — пока **generic units** (считаем вызовы/запросы). Прайс/деньги — отдельный
   слой позже, поверх метеринга. Не блокирует.
3. **Где живёт** — **внутри Ванги**, корп-инфра, расширяем текущий шлюз. Отдельный облачный
   инстанс не нужен. Корп-правило «облачные LLM нельзя» остаётся (локальная phi4); DaData/Яндекс —
   РФ-API, корп-допустимо.
4. **Комплаенс** — ок (договорные вопросы с владельцами API/потребителями решаемы).

## Что уже есть (каркас ~⅔)
- **Сейф ключей/интеграций** — `provider_settings`, kind-based (`llm` ×15, `ocr` ×2, `dadata` готов).
- **Шлюз** — `/v1/*` проксирует LLM (chat/embeddings/Anthropic) **и DaData** (роут есть, спит за флагом),
  auth по PAT.
- **Потребители** — `users.kind=service` (сервис-аккаунты) + PAT-токены + мультитенант `organizations`.
- **Метеринг** — `llm_gateway_usage` (`caller / alias / model / tokens / latency / status / started_at`).

## Целевая модель данных
Ось учёта: **потребитель × коннектор × usage(units) × время**.
- **Потребитель** = инстанс/проект как отдельная сущность (организация ИЛИ tagged сервис-аккаунт)
  со своим PAT. Каждый вызов шлюза однозначно атрибутируется потребителю.
- **Коннектор** = внешний API (`llm`, `dadata`, `yandex_maps`, …) с центральным ключом в сейфе.
- **Usage** = обобщить `llm_gateway_usage` → generic: `consumer_id, connector, units, unit_kind
  (tokens|calls|geocodes…), status, latency, ts`. LLM кладёт tokens, DaData/Яндекс — calls.

## Четыре столпа — что достроить
| Столп | Есть | Достроить |
|---|---|---|
| Управление | сейф ключей, UI Провайдеры/Системы | экран «Интеграции»: коннектор + ключ + вкл/выкл + кому доступно |
| Контроль | PAT + сервис-аккаунты + org | per-коннектор/per-потребитель доступ, rate-limit, квоты (EXT-LLM-PROXY-C) |
| Мониторинг | usage-лог, здоровье вебхуков | дашборд: вызовы/ошибки/latency по коннектору и потребителю + health API |
| Учёт/биллинг | usage-лог (только tokens) | generic units, агрегация per-потребитель → отчёт «кто сколько»; деньги — позже |

## Коннекторы (порядок)
1. **DaData** — роут есть, **добить активацию** (ключ в сейф + флаг). Реальный спрос: подразделение по ИНН.
2. **Яндекс.Карты** — новый коннектор: геокодинг + маршруты (логистика — расстояние/время плеча).
3. Дальше по спросу: 1С/Bitrix (штатные API), курсы валют, проверка контрагентов и т.п.

## Статус реализации (2026-06-25 — собрано end-to-end, ключи позже)
- ✅ **Ф1 backbone** — `gateway_connectors` (реестр+caps) + `gateway_consumer_budgets` + обобщённый
  метеринг (`llm_gateway_usage` += connector/units/unit_kind) + `checkConsumerQuota`. Мигр. `…0004`. В проде.
- ✅ **Ф2 коннектор Яндекс.Карты** — `/v1/maps/geocode` + `/v1/maps/route` (geocode-maps/routing.yandex,
  apikey в query), спит за флагом (503 без ключа), метерится. Мигр. `…0005` (kind=yandex_maps). В проде.
- ✅ **DaData-коннектор** — `/v1/dadata/*`, спит за флагом (был ранее). В проде.
- ✅ **Управляющий слой** — `/api/v1/gateway/{connectors,budgets,usage}` (super_admin): CRUD коннекторов
  (enabled/caps), бюджеты consumer×connector, агрегаты usage. enforceQuota в 4 точках шлюза за флагом
  `LLM_GATEWAY_QUOTA_ENABLED` (default off) + fail-open. В проде.
- ✅ **Ф3 UI** — `/settings/integrations` (super_admin): коннекторы (тумблер+caps), бюджеты, дашборд
  usage (период/потребитель/коннектор + спарклайн). В проде.
- ⏭ **Осталось (нужен owner/ключи):** вписать реальные ключи Яндекс/DaData (через UI Провайдеры,
  kind=yandex_maps/dadata) → включить коннекторы (PATCH enabled=true) → выставить `daily_cap` под
  реальный лимит ключа → завести per-consumer PAT'ы (СЛАЙ-инстансы / подразделения) + бюджеты →
  включить `LLM_GATEWAY_QUOTA_ENABLED=true`. Уточнить точный router-эндпоинт Яндекса при наличии ключа.
- ⏭ **Позже (Ф4/Ф5):** месячные квоты, прайс/деньги (когда определим единицу), кэш геокода,
  multi-instance management UI (MTI-1).

## Первый шаг (рекомендация)
**Ф1**, потому что под него уже есть реальный спрос (подразделение хочет DaData/ИНН): обобщить
метеринг + завести потребителя как сущность + активировать DaData. Это спина всего хаба — на неё
дальше навешиваются коннекторы, мониторинг и учёт.
