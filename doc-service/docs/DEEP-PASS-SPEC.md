# DEEP-PASS — глубокий разбор нераспознанного остатка

> **Статус:** согласовано с владельцем 2026-07-15 (развилки 1–4 решены).
> **Фаза 1** реализуется сразу; фазы 2+ — бэклог.

## Проблема

Воронка жёсткая: документ либо попадает в один из ~49 рабочих типов и
разбирается по схеме, либо — тупик. Для «не опознан» извлечение вообще не
запускается (`orchestrator`: extract гейтится на `documentType`), поля
пустые, статус `needs_review`. Оператор открывает карточку и не видит
ничего. Скриншоты переписок, чужие типы, плохие сканы — падают без пользы,
а часть (OCR-refusal) ещё и гоняется 3 ретрая впустую.

## Решение: второй ярус обработки

**Ярус 1 (существующий):** быстрый разбор по рабочему каталогу типов.

**Ярус 2 (deep-pass), только для остатка.** Триггер — **автоматический**
(решение владельца), гейт `DEEP_PASS_ENABLED`:

| Триггер | Когда |
|---|---|
| `classify_unknown` | классификатор (включая §P2-2 VLM-фолбэк) не дал тип |
| `ocr_refused` | vision-модель отказалась читать изображение (раньше — fail + 3 ретрая) |

Шаги deep-pass:

1. **Чтение.** Если OCR-текста достаточно (≥ `DEEP_PASS_MIN_TEXT`) —
   текстовый LLM-вызов (первые `DEEP_PASS_TEXT_CHARS` символов — больше,
   чем 2500 у классификатора). Иначе — VL-модель по картинке первой
   страницы (локальная, наружу ничего не уходит).
2. **Широкая классификация.** Фиксированный словарь ~18 глобальных
   категорий (`broad-catalog.ts`) — решение владельца: фиксированный,
   рабочий каталог закрывает ~90% потока, широкий — для остатка.
3. **Резюме.** 2–3 предложения «что это и о чём» + язык документа.
4. **Маппинг на рабочий каталог.** Модель может вернуть slug рабочего
   типа (видит каталог целиком) → **возврат в конвейер**: обычное
   извлечение по схеме этого типа (re-run pipeline с hint,
   `classification.method='deep_pass'`, confidence 0.7 — гейт
   `classify_uncertain` применяется честно).

Вердикты: `mapped` (наш тип, вернули в конвейер) · `foreign_document`
(полезный, но чужой — широкая категория + резюме) · `not_a_document`
(скриншот/мусор — отдельная честная категория, решение владельца).

## Куда пишется результат

`extracted._deep` (по паттерну служебных `_issues`/`_enrichment`):

```json
{
  "broad_type": "correspondence_screenshot",
  "broad_label": "Скриншот переписки",
  "language": "zh",
  "summary": "Скриншот переписки в мессенджере …",
  "catalog_slug": null,
  "verdict": "not_a_document",
  "via": "vision"
}
```

- `_deep` не считается бизнес-полем (`countBusinessFields` пропускает `_*`)
  — гейты empty-extract/needs_review не ломаются.
- Уходит потребителям внутри `extracted` (webhook) — дополнительная
  информация, контракт не меняется.
- UI: бейдж широкой категории (amber) / «Не документ» (slate) в колонке
  типа, tooltip = резюме.

## ПДн (152-ФЗ)

- `broad_type=id_document` → резюме заменяется константой «Удостоверение
  личности — содержимое не сохраняется», маппинг на каталог не выполняется.
- Любое резюме проходит `scrubPassportPatterns` (belt-and-suspenders).
- Vision — только локальный провайдер (`withVisionProvider`), как §P2-2.

## Ретраи детерминированных ошибок (попутный фикс)

`OcrRefusedError` и `no OCR engine available for mime type` — детерминированы
(тот же файл → тот же результат). Worker оборачивает их в `UnrecoverableError`:
1 попытка вместо 3. При включённом deep-pass refusal вообще не фейлит job —
уходит на ярус 2.

## Конфиг

| env | default | смысл |
|---|---|---|
| `DEEP_PASS_ENABLED` | `false` | гейт всего яруса |
| `DEEP_PASS_TEXT_CHARS` | `8000` | сколько текста уходит в text-путь |
| `DEEP_PASS_MIN_TEXT` | `300` | меньше → vision-путь по картинке |

## Экономика

Deep-pass идёт только по остатку (обычно единицы % потока): +1 LLM-вызов
(text) или +1 VL-вызов (vision), +1 extract при возврате в конвейер. Всё
локально. Расход виден в `llm_usage`/₽-колонке как у остальных стадий.

## Фазы

- **Фаза 1 (этот документ, реализована):** триггеры unknown+refusal,
  широкий словарь, резюме, возврат в конвейер, `_deep` + UI-бейдж,
  UnrecoverableError для детерминированных ошибок.
- **Фаза 2 (бэклог):** триггер по `extract_empty` (тип известен, полей 0);
  multidoc-сегменты без типа; открытое извлечение ключ-значение для
  `foreign_document`; агрегатная страница «что приходит мимо каталога»
  (data-driven расширение каталога).
- **Фаза 3 (бэклог):** отдельный статус/фильтр «не документ» в API-контракте
  (schema_version bump), автозакрытие мусора без ревью — после того как
  вердиктам начнут доверять.
