> Источник: многоагентный разбор 2026-07-10 (understand→design→judge→synthesize) + адверсариал-ревью реализации. P0 задеплоен.

# parsdocs · Пересборка раздела «Настройки» — рекомендация

## 1. TL;DR

Свести пять табов в **три: «Подключения» / «Инстанс» / «Пробный разбор»**. Ядро — экран «Подключения»: сверху строка-баннер «что активно на этом сервере», ниже — карточки по одному реальному сервису, сгруппированные по назначению (Разбор / Распознавание / Обогащение / Геокод). Клик по карточке разворачивает **всё про сервис в один вертикальный скролл**: подключение → статус → расход → история. Каждая интеграция = одна карточка, а не две строки в двух таблицах.

Почему лучше текущего: сейчас один сервис (напр. Яндекс Vision) размазан на `provider_settings` + `gateway_connectors` + env, а «выключить Яндекс» физически живёт на двух тумблерах с обратными fail-режимами — отсюда класс ошибки «выключил не тот». Новая IA берёт **способность-центричную группировку** (лучший ответ не-технику «какой из трёх Яндексов и где»), кладёт её в **связочно-центричный контейнер** (даёт дом будущим не-пайплайновым интеграциям — 1С/вебхуки/SSO), и вводит **жёсткий контракт «показываем только то, что реально читает резолвер»** — никаких контролов-пустышек.

**Каркас** — победившая по операторской и инженерной линзам Схема 2 (capability-first). **Привои:** «всё про сервис в одном скролле» (Схема 1), future-proof контейнер и честный Vision (Схема 4), условный рендер секций + usage-by-model (Схема 3).

## 2. Новая навигация: было 5 → стало 3

| Было (5 табов) | Стало | Судьба |
|---|---|---|
| Общие | → **Инстанс** | Остаётся read-only снимком (storage/ретеншены/endpoints/флаги секретов). Минус то, что переехало в статус-строки карточек. Логаут/сессия **вынесены в меню профиля** (это никогда не было настройкой). |
| Провайдеры/модели | → **Подключения** | **Вливается.** CRUD-строки становятся карточками, сгруппированными по назначению. |
| Интеграции *(super_admin)* | → **Подключения** | **Вливается.** Коннекторы (тумблер) → статус карточки; Расход → секция карточки; Бюджеты → скрыты до энфорса. Гейт super_admin переезжает с целого таба на **per-action** (бэкенд `requireSuperAdmin` не трогаем). |
| Журнал аудита | → **растворяется** | Глобального таба **нет**. Переименован в «История изменений», живёт секцией внутри карточки + демотированной ссылкой на «Инстансе». (§5) |
| Тест-лаборатория | → **Пробный разбор** | Остаётся; честное имя; deep-link из карточки LLM «Проверить на документе». |

**Итог: 3 таба.** «Подключения» (единственная write-поверхность и главный экран) · «Инстанс» (read-only снимок) · «Пробный разбор» (единственный action-инструмент).

**Почему контейнер «Подключения», а не «Способности» (правка каркаса).** Схема 2 приклеивала верх к фиксированной онтологии из 4 пайплайн-способностей — по extensibility-линзе это структурная стена: первая же не-пайплайновая интеграция (1С-экспорт, вебхук, нотификации) остаётся без дома. Решение: **способность-группировка сохраняется как заголовки секций внутри «Подключений»**, но контейнер называется по связи, а не по способности. Добавить новую группу («Экспорт и уведомления») = дописать секцию, а не редизайн. Пустые группы не рендерим — дом готов, но появляется вместе с первой реальной интеграцией.

Legacy-редиректы обязательны: `/settings/providers → /settings/connections`, `/settings/integrations → /settings/connections`, `/settings/audit → /settings/connections?history=all`, `/settings/lab → /settings/trial`.

## 3. Поэкранно

### 3.1 «Подключения» (default) — главный экран

**Баннер инстанса** (одна строка сверху, всегда виден) — из `GET /settings` + `findDefault('llm')` + `extra.backend`, без нового бэка. Критично при двух серверах/VPN — за 5 секунд отвечает «где я и что активно»:

**Каталог карточек**, сгруппированный по назначению. Карточка в свёрнутом виде: имя · бейдж `локально/облако/GPU` (вычислен, не флаг) · **один реальный статус-индикатор** · ключевой факт (ключ/модель/folder есть-нет) · **расход за сегодня на лице**. Значок ⚠ — только где реальный риск (ПДн в облако).

```
┌─ ПОДКЛЮЧЕНИЯ ─────────────────────────────────────────────────────┐
│ Инстанс: АША · Разбор — YandexGPT (облако) · Распознавание —        │
│ Yandex Vision ✓ · Обогащение — DaData ✓ · Геокод — Карты —          │
├───────────────────────────────────────────────────────────────────┤
│ МОДЕЛИ РАЗБОРА (LLM)               [фильтр: все типы ▾] [+ добавить] │
│ ┌───────────────────────────┐ ┌───────────────────────────┐        │
│ │ ★ YandexGPT      [облако]  │ │   Qwen 2.5 VL   [локально] │        │
│ │ активная модель разбора    │ │   Ollama · достижима ✓     │        │
│ │ gpt://…/latest · ключ ✓    │ │   qwen2.5vl:7b · ключ —    │        │
│ │ сегодня: 1.2k вызовов      │ │   сегодня: 0               │        │
│ └───────────────────────────┘ └───────────────────────────┘        │
│ РАСПОЗНАВАНИЕ (OCR)                                                 │
│ ┌───────────────────────────┐ ┌───────────────────────────┐        │
│ │ ● Яндекс Vision  [облако]  │ │ ▪ Встроенный OCR           │        │
│ │ ⚠ сканы уходят в облако    │ │ tesseract+pdf-parse · env  │        │
│ │ ключ ✓ · folder ✓          │ │ read-only                  │        │
│ └───────────────────────────┘ └───────────────────────────┘        │
│ ОБОГАЩЕНИЕ / ГЕОКОД                                                 │
│ ┌───────────────────────────┐ ┌───────────────────────────┐        │
│ │ ● DaData         [облако]  │ │ ○ Яндекс.Карты   [облако]  │        │
│ │ ИНН→ЕГРЮЛ · ключ ✓         │ │ адрес→гео · ключ —         │        │
│ └───────────────────────────┘ └───────────────────────────┘        │
└───────────────────────────────────────────────────────────────────┘
```

- **Фильтр по типу возвращён** (Схема 1 его выпилила — прямой extensibility-страйк): группа «Модели разбора» растёт (VANGA-LLM-2: +5–10 строк), карточки без поиска кучкуются. Модели держим **сворачиваемой группой** — N моделей не засоряют верх.
- **Бейдж «★ активная модель разбора»** — только у `is_default` строки LLM. Это визуальный якорь per-instance-выбора.
- **Mental model:** «каждый внешний сервис — одна карточка; вижу назначение, включён ли, сколько тратит; не надо помнить, что ключ в „Провайдерах“, а рубильник в „Интеграциях“».

### 3.2 «Инстанс» (было «Общие»)

Read-only снимок того, что **про инстанс, а не про сервис:** идентичность (Taipit/Аша), storage/sweeper'ы/ретеншены (включая ретеншен истории), endpoints (Swagger/health/metrics), флаги секретов configured/not-set. Здесь же — **демотированная ссылка «История изменений»** (рядом с её ретеншен-конфигом, закрывая шов «объект и его настройка на разных вкладках»). Логаут отсюда убран.

### 3.3 «Пробный разбор» (было «Тест-лаборатория»)

Без изменений функции. Переосмыслен в текстах как **сравнение исполнителей разбора**: «прогони этот УПД через Qwen, потом через YandexGPT, сравни confidence» (`metadata._force_provider_id` уже это делает) — это ровно сценарий Taipit-vs-Аша на одном документе. Кнопка запуска — `manager+`.

## 4. Единая карточка интеграции (ядро запроса)

Клик по карточке → разворачивается панель, **всё про сервис в один вертикальный скролл** (Схема 1), без master-detail-глубины. Секции рендерятся **условно** (Схема 3): нет коннектора → нет «Расхода»; «Лимиты» скрыты всегда (§7).

```
┌─ Яндекс Vision · Распознавание ──────────────────────────── ✕ ──┐
│ [ Облачное распознавание:   ●ВКЛ ]     ← ОДИН рубильник         │
│ ⚠ При включении сканы (в т.ч. ПДн) уходят в облако Яндекса.     │
│   Выключение блокирует отправку сразу (egress-гейт).           │
│                                                                │
│ ПОДКЛЮЧЕНИЕ                                                    │
│   API-ключ        ••••1234   [изменить]                        │
│   folder_id       b1g…xyz         ← было только в сыром JSON    │
│   (Base URL и модель заданы платформой — не показаны)          │
│                                                                │
│ РАСХОД (сегодня / 30 дней)                                     │
│   страниц: 340 · ошибок: 2      ▁▂▅▃▆▂▁                        │
│   ⓘ лимиты пока не режут вызовы — показан только факт          │
│                                                                │
│ ИСТОРИЯ ИЗМЕНЕНИЙ                                              │
│   10.07 admin · ключ обновлён (••••→••••)                     │
│   ⓘ смена тумблера/лимита пока не логируется (P1)             │
└────────────────────────────────────────────────────────────────┘
```

**Как «провайдер + рубильник + лимиты + расход» встают на одно место:**
- **Подключение** = `provider_settings` (форма `ProviderEditor` переиспользуется 1:1). Для LLM — дружелюбные поля Бэкенд/Upstream/model/reasoning + **чекбокс `vision`**; сырой JSON — под «для продвинутых».
- **Статус** = один рубильник (маппинг ниже) + для LLM «★ Сделать активной моделью» (`set-default`, hot-swap).
- **Расход** = `GET /gateway/usage?connector=…` — вызовы/единицы/ошибки + спарклайн. Read-only, честные чтения.
- **История** = `audit_log` фильтром `entity_id`.

**Маппинг «один честный рубильник → что пишет → что делает рантайм»:**

| Карточка | Рубильник пишет | Реальный эффект | Что убрано как фейк |
|---|---|---|---|
| **LLM-модель** | `is_active` + `set-default` | `findDefault('llm')` резолвит исполнителя без рестарта | connector `llm.enabled`/caps (пайплайн игнорит) |
| **Яндекс Vision** | **`connector.enabled`** (egress) | fail-closed: выключение сразу блокирует отправку в облако | второй тумблер `is_active` как «power» |
| **Встроенный OCR** | нет тумблера | каскад из env; управляется порогами на «Инстансе» | фейковый вкл/выкл |
| **DaData** | `is_active` + `set-default` | `findDefault('dadata')` → Token | `base_url`/`model` (env), `secret_key` (не читается) |
| **Яндекс.Карты** | `is_active` + `set-default` | `findDefault('yandex_maps')` | `base_url`/`model` (env) |
| **Разбор (способность)** | тумблера **нет** | парсинг обязателен; «выкл» = выбрать исполнителя `stub` | fake kill-switch |

**Развязка узла Vision (crux, где линзы конфликтовали — привожу синтез).** Сейчас два тумблера с обратными fail-режимами: `is_active` (fail-**open** на env-ключ) и `enabled` (fail-**closed** на egress). 
- Операторская линза: два тумблера как первичная операция проваливают 5-сек-тест. 
- Инженерная линза: один атомарный тумблер поверх двух гейтов без серверной транзакции воссоздаёт баг «выключил не тот».

**Решение — не «атомарно писать оба» и не «два равноправных», а привязать ЕДИНСТВЕННЫЙ рубильник к ОДНОМУ гейту — egress (`connector.enabled`).** Именно он fail-closed: выключен → сканы физически не уходят, это ровно тот «kill-switch», которого боится основатель. `is_active` — это «какой ключ (UI или env) использовать, когда egress разрешён», а **не** «включено ли»; поэтому он спускается в «Подключение» как деталь ключа, а не как power-тумблер. Так у видимого рубильника нет второго скрытого гейта, который он якобы контролирует → оба дилбрейкера сняты: **не нужен новый атомарный endpoint** (пишем существующий `PATCH connector.enabled`) **и** оператор гасит Яндекс одним действием. Предупреждающий текст из Схемы 4 сохраняем, чтобы единый тумблер не стал молчаливым чёрным ящиком. (Опционально P2: синхронизировать `is_active`↔`enabled` на бэке.)

## 5. Журнал аудита — судьба

**Глобальный таб удаляется. «Журнал аудита» → «История изменений»** — снимаем ложное обещание комплаенс-аудита: по факту логируются CRUD ровно двух сущностей (`provider_setting`, `document_type`), актор всегда `'admin'` (общий Bearer). Это не «кто именно», а change-log.

- **Первичный дом — секция «История изменений» внутри карточки** сервиса (фильтр `entity=provider_setting&entity_id=<id>`; URL-state уже поддерживает шаринг). Оператор ищет «кто менял ключ Vision» именно там.
- **История типов документов** уезжает на экран «Типы документов» (её естественный дом) той же секцией.
- **Демотированный глобальный вид** — ссылкой с «Инстанса», рядом с ретеншеном (закрывает шов). Тот же `AuditLog.tsx` без фильтра.
- **Актор:** показываем честно «admin (общий токен)», не имитируем «кто именно» до per-user токенов.
- **Обязательная плашка охвата:** «изменения тумблера/лимита (`gateway_connectors`/`budgets`) сюда пока не попадают». Это латентный gap **во всех схемах** — `audit_log` не покрывает gateway-таблицы. Пока не расширим (P1), per-card история неполна — и мы это подсвечиваем, а не прячем.

## 6. Локальные модели + per-instance (Taipit=qwen / Аша=Яндекс)

**Механизм уже существует — ничего в рантайме не меняем.** Два инстанса = две отдельные БД, значит **свой `is_default('llm')` на каждой**. Свитчер инстансов не нужен — нужен честный «активный движок здесь» (это и есть баннер §3.1).

1. **Дом локальной модели.** Мёртвый seed `qwen-local` (`is_active=false`, ни с чем не связан) заменяем **живой строкой-профилем:** реальный тег `qwen2.5vl:7b`, `extra.backend='openai_compat'`, `extra.upstream_base_url='http://ollama:11434/v1'`. Все три поля уже читаются резолвером (VANGA-LLM-2) — теперь это поля секции «Подключение», а не только сырой JSON.
2. **Бейдж «локально/облако/GPU»** — **вычисляется** из `extra.backend`+`upstream_base_url` (upstream на внутренний хост = локально), а не новый флаг.
3. **Per-instance-выбор = какая LLM-строка `is_default`.** Taipit: `★ Qwen 2.5 VL`. Аша: `★ YandexGPT`. Переключение = `set-default` в пуле LLM (рабочий контрол, `dynamicLlm.invalidate()`, без рестарта, без SSH). Баннер сверху пишет активный движок для текущей БД.
4. **Честная живость** — реальный пинг через существующий `GET /providers/status` (проксирует inference→Ollama): «достижима ✓ / недостижима ✕». **НЕ врём**, какой тег реально `pull`-нут в Ollama (`OLLAMA_PULL`/`OPENAI_MODEL` живут в env инференса вне БД) — показываем «заявлено `qwen2.5vl:7b`», опц. «загружено: <из inference /api/tags>». Никаких зелёных галок без реального probe: если тег не загружен, зелёная достижимость + падение на первом вызове отразится в «Расход»→ошибки.

## 7. Антипаттерн «фейковые контролы» — как IA это гарантирует

**Жёсткий дизайн-контракт (из §0 Схемы 2):** каждый видимый контрол обязан маппиться на конкретное чтение резолвера (`findDefault` / `extra.backend` / `folder_id` / `api_key`). Что рантайм не читает — **не рисуем как контрол**. Практически:

**Прячем мёртвое:**
- `secret_key` (dadata) — нигде не читается, «задел на будущее» → убрать из формы.
- `base_url`/`model` для dadata/yandex_maps/ocr — env-driven, декоративны → скрыть.
- `set-default`/«Исполнитель» для OCR — `findDefault('ocr')` не зовётся нигде; запишет `is_default`, вернёт 200, ничего не изменит → **убрать fake-dropdown**, заменить утверждением «Облачный движок: Yandex Vision».
- Кнопку «Тест» для OCR (пингует пустой base_url, бесполезна) — заменить на «Проверить на документе» → Пробный разбор.

**Выводим живое-но-скрытое** (читается рантаймом, но правилось только сырым JSON):
- `vision` (чекбокс) — читается `findActiveVision`, но в UI/zod отсутствует.
- `extra.folder_id` (текст) — читается OCR-резолвером.
- Добавить оба в zod `Create/PatchBody`, `ProviderApi`, UI-тип.

**Лимиты/бюджеты — скрываем редакторы целиком.** `checkConsumerQuota` написана, но **не вшита в роуты** → `daily_cap`/`daily_budget` сохраняются, но 101-й вызов не блокируют. Показать редактируемый «cap=100» = ровно та запретная пустышка (не-технический оператор дисклеймер не прочитает). Оставляем **только read-only «учтено сегодня»**. Редакторы вернутся автоматически, когда энфорс будет вшит (P2) — не раньше.

**Разбор без kill-switch.** Документ обязан парситься; «выключить разбор» = выбрать исполнителя `stub`. Отсутствие тумблера — это честность, а не недоделка.

Эта двусторонняя разфейковка (спрятать мёртвое + вывести живое) — общий вывод всех трёх линз и всех четырёх схем.

## 8. План внедрения фазами

**Переиспользуется целиком (бэкенд не трогаем, схема БД не меняется):** `provider_settings` CRUD + `/set-default` + `/test`; `gateway_connectors` PATCH (`enabled`); `GET /gateway/usage?connector=`; `audit_log` read (фильтр `entity_id`); `GET /providers/status`; `GET /settings`; компоненты `ProviderEditor`, diff-viewer из `AuditLog.tsx`, `DailySparkline`, `Toggle`, `ExecutorPicker`/`useSetDefaultProvider`; TestLab job-flow. Слияние — **клиентский join** `provider_settings ⨝ gateway_connectors` по `kind↔provider_kind`/`slug`, не новая сущность.

**P0 — быстрый выигрыш (почти весь объём — фронт на существующих ручках):**
- Экран «Подключения»: контейнер каталога карточек + группировка по назначению + баннер инстанса + панель «всё в один скролл» (композиция уже существующих компонентов).
- Нав 5→3, роуты, breadcrumbs, legacy-редиректы. Гейт super_admin с таба → на per-action.
- Разфейковка (§7): скрыть мёртвое, вывести `vision`+`folder_id` в форму/zod, спрятать редакторы лимитов (оставить «учтено сегодня»), убрать OCR-dropdown.
- Vision: один рубильник на `connector.enabled` + предупреждение; `is_active` → деталь ключа.
- Аудит → «История изменений»: per-card секция + демотированная ссылка + плашка охвата.
- Живая строка локальной модели вместо seed `qwen-local` (правка seed/миграции). Бейдж backend вычисляемый.
- Логаут → меню профиля. Фильтр по типу в группе LLM.
- **Объём: средний, преимущественно фронт.** Самодостаточно — уже даёт founder'у честную картину.

**P1 — честностные бэкенд-доработки (закрывают «врущие умолчания»):**
- Расширить `audit_log.entity` на `gateway_connectors`+`gateway_consumer_budgets` + вызвать `append()` в `gateway-admin.ts` → per-card история полна. **Средне, обязательно.**
- Usage `groupby model` → per-model расход (иначе сравнение моделей в Пробном разборе показывает общий connector-агрегат `'llm'`). **Low-med.**
- (Опц.) серверный `GET /resources`-агрегатор: централизует join + спец-кейс Vision (`id='yandex-vision'`) в одном месте → новая интеграция вписывается правкой одного файла, а не размазанным клиентским join'ом. **Держать опциональным** — client-join достаточен для P0, но при росте интеграций мягкий строковый join (уже сломан для Vision: `kind='ocr'` vs `provider_kind='yandex_vision'`) тихо разъезжается; агрегатор — страховка расширяемости.
- (Опц.) Ollama loaded-tag через inference `/api/tags`.

**P2 — превращает спрятанное в рабочее / крупное:**
- Вшить `checkConsumerQuota` в роуты → лимиты реально режут → **тогда** показываем редакторы (сейчас скрыты). Предсуществующий долг, вне scope IA.
- Per-user actor-токены → аудит отвечает «кто именно».
- (Опц.) синхронизация `is_active`↔`enabled` для Vision.
- **Триггер следующей абстракции (задокументировать, не строить сейчас):** «Каталог + Инстансы» с общим governance ключей становится правильным при **≥3 инстансах** или требовании единой политики. Сейчас (2 инстанса, раздельные БД) — over-engineering; карточки = «каталог», набор `is_default` = «профиль инстанса».

## 9. Риски и открытые вопросы (честно)

1. **Vision — привязка одного рубильника к `enabled` меняет ментальную модель тех, кто сознательно держал `is_active=false` ради env-fallback ключа.** Нужен аккуратный дефолт и релиз-нота. Плюс остаётся кейс `enabled=true & is_active=false` → идёт env-ключ; поэтому `is_active` показываем в «Подключении» как «источник ключа», а не как «вкл/выкл». Проверить на реальных данных обоих инстансов до релиза.
2. **Per-card «История» неполна до P1.** `audit_log` не логирует gateway-таблицы → смена тумблера/лимита невидима. Смягчено плашкой охвата, но это реальный gap — если P1 не сделать, история «врёт умолчанием».
3. **Расход не бьётся по модели до P1.** Несколько LLM-строк, пишущих в один connector `'llm'`, показывают общий агрегат — вводит в заблуждение ровно в сценарии сравнения моделей (который переносим в Пробный разбор).
4. **Достижимость ≠ загруженность тега.** `/providers/status` покажет живой Ollama, но не что нужный тег `pull`-нут → возможен зелёный статус + падение на первом вызове. Закрывается «Расход»→ошибки, но карточка может выглядеть оптимистичнее реальности.
5. **Осознанная асимметрия карточек** (Разбор богат/переключаем, OCR — фиксированный каскад, DaData/Карты — один ключ). У оператора нет единой модели «одна карточка = один паттерн»; каждую надо чуть доучить. Это честная UX-цена честности — не маскируем реальную разницу в том, что читает код.
6. **Скрытые редакторы лимитов** снижают «полноту» ради честности. Если parsdocs станет мульти-потребительским на одном инстансе (перепродажа шлюза), гранулярность `consumer×connector` временно потеряна — вернётся в P2 вместе с энфорсом.
7. **Нет единого пульта над двумя инстансами** — Taipit и Аша конфигурятся раздельно (разные БД). Кросс-инстанс-обзор вне рамок; триггер перехода на «Каталог+Инстансы» задокументирован (≥3 инстанса).
8. **Смешанная авторизация внутри одного экрана** (ключ — admin; connector-действия — super_admin; «Прогнать» — manager+) сложнее двух раздельно-гейченных табов. Гейт переносим на per-action поверх неизменного бэкового `requireSuperAdmin`, но это требует аккуратного QA, чтобы не сделать дырку.
9. **IA не чинит корневые двусмысленности данных** (тройной источник ключей env/`provider_settings`/`llmGateway.*`, precedence `env > provider_settings`, отсутствие FK). Красивый экран может замаскировать, что реально решает `.env`. Это отдельный долг данных, не IA.
