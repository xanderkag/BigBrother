# ТЗ — внутренний биллинг: себестоимость, отчёты, бюджеты

**Дата:** 2026-07-22 · **Статус:** ТЗ, не реализовано
**Решения владельца (2026-07-22):** внутренний учёт (счетов наружу НЕ делаем) ·
бюджет предупреждает, не блокирует · ставки переносятся на провайдера.

---

## 1. Зачем

Сейчас на вопрос «почему потратили 600 ₽» можно ответить только SQL-запросами руками.
`cost_rub` считается по каждой задаче, но **агрегации нет ни одной**, бюджетов нет, а
расход через LLM-шлюз не считается в деньгах вообще. Цель — закрыть это: видеть расход
в разрезах, ловить всплески порогом, и уметь объяснить любую цифру постфактум.

**Границы (что НЕ делаем):** тарифные планы, наценка, счета и акты наружу, балансы и
предоплата, блокировка при превышении. Модель данных проектируется так, чтобы это
навесилось позже без ломающей миграции, но сейчас не реализуется.

## 2. Что уже есть

| Компонент | Файл / объект | Состояние |
|---|---|---|
| Расчёт per-job | `pipeline/cost.ts` → `computeJobCost()` | 4 ставки, результат в `jobs.cost_rub` + `cost_estimate` |
| Ставки | `config.ts:348` (`config.cost`) | 4 числа, глобальные на инстанс, env-переопределяемые |
| Сырьё расхода | `jobs.llm_usage` (prompt/output tokens, calls), `jobs.ocr_pages` | есть |
| Учёт шлюза | таблица `llm_gateway_usage` | caller, alias, model, токены, latency, status — **денег нет** |
| Разрезы | `organizations`, `projects`, `jobs.organization_id`, PAT-ключи | есть |
| Отчёты | — | **отсутствуют** |
| Бюджеты | — | **отсутствуют** |

## 3. Ключевая проблема — ставки должны висеть на провайдере

`config.cost` зашит под Yandex (0.13 / 1.22 / 0.2 / 0.3 ₽). Но в одном инстансе
одновременно работают Yandex (облако, платим деньгами), локальный vLLM (платим
железом) и Anthropic (облако, платит в USD). Одна пара ставок на инстанс врёт, как
только провайдеров больше одного — а их уже больше.

Второе: **ставки меняются**. Если хранить только итоговую сумму, пересчитать историю
и объяснить её нельзя; если хранить только текущие ставки на провайдере — пересчёт
старых задач применит новые ставки и исказит прошлое.

**Решение:** ставки живут на провайдере, а **применённая ставка фиксируется в задаче**
(снимок в момент расчёта). Это даёт и честный мульти-провайдерный расчёт, и аудит.

## 4. Модель данных

### 4.1 Прайс-лист на провайдере

`provider_settings` += колонка `rates JSONB NULL`:

```json
{
  "currency": "RUB",
  "cost_basis": "vendor",
  "llm_input_per_1k": 0.2,
  "llm_output_per_1k": 0.3,
  "ocr_page": 0.13,
  "ocr_page_table": 1.22
}
```

- `currency` — валюта провайдера (`RUB` / `USD`). Anthropic считает в USD.
- `cost_basis` — **важное поле для отчётов**:
  - `vendor` — реальные деньги наружу (Yandex, Anthropic);
  - `amortized` — своё железо, ставка это внутренняя переводная цена (локальный vLLM);
  - `free` — не тарифицируется (stub, dev).

  Отчёт обязан уметь показать «деньги наружу» отдельно от «нагрузка на своё железо» —
  это разные вещи для владельца, смешивать их нельзя.
- Ставка отсутствует (`rates = NULL`) → расход считается неизвестным, задача помечается
  `cost_estimate = true`. **Молча считать нулём запрещено** — ноль неотличим от «бесплатно».

### 4.2 Снимок расчёта в задаче

`jobs` += колонка `cost_breakdown JSONB NULL` (`cost_rub` остаётся как денормализованный
итог для сортировки и списка):

```json
{
  "currency": "RUB",
  "total": 3.20,
  "estimate": false,
  "computed_at": "2026-07-22T10:15:00Z",
  "lines": [
    { "kind": "ocr_page_table", "provider_id": "yandex-vision",
      "qty": 2, "unit": "page", "rate": 1.22, "sum": 2.44, "cost_basis": "vendor" },
    { "kind": "llm_input", "provider_id": "yandex-ai-studio",
      "qty": 3800, "unit": "token", "per": 1000, "rate": 0.2, "sum": 0.76, "cost_basis": "vendor" }
  ]
}
```

Правила:
- строка на каждую пару (провайдер × вид расхода);
- `rate` — ставка **на момент расчёта**, не ссылка на провайдера;
- пересчёт задним числом (`reprocess`) перезаписывает `cost_breakdown` целиком и ставит
  новый `computed_at`;
- сумма `lines[].sum` обязана сходиться с `total` (проверяется тестом).

### 4.3 Стоимость в шлюзе

`llm_gateway_usage` += `cost_rub NUMERIC NULL`, `cost_breakdown JSONB NULL` — тот же
формат. Ставка берётся у провайдера, на который смотрит алиас. Без этого второй канал
расхода (SLAI ходит к нам за чатом) остаётся невидимым.

### 4.4 Экономия от кэша

Повторная загрузка того же файла отдаёт закэшированную задачу и стоит 0. Сейчас это
нигде не отражено, хотя именно кэш — главный рычаг экономии (в июльском инциденте
корпус гоняли трижды).

В `cost_breakdown` кэш-хита:

```json
{ "currency": "RUB", "total": 0, "estimate": false,
  "cache_hit": true, "would_cost": 3.20, "source_job_id": "uuid" }
```

Отчёт показывает «сэкономлено за период» = сумма `would_cost` по кэш-хитам.

### 4.5 Бюджеты

Новая таблица `budgets`:

| Колонка | Смысл |
|---|---|
| `id` | PK |
| `scope` | `instance` \| `organization` \| `project` |
| `scope_id` | UUID организации/проекта; NULL для `instance` |
| `period` | `month` (на старте только месяц) |
| `limit_rub` | порог, с которого предупреждаем |
| `thresholds` | массив долей, дефолт `[0.5, 0.8, 1.0]` |
| `hard_limit_rub` | **NULL, зарезервировано** — сюда встанет блокировка, когда понадобится |
| `is_active`, `created_at`, `updated_at` | — |

Новая таблица `budget_events` — факт пересечения порога (`budget_id`, `threshold`,
`period_start`, `spent_rub`, `limit_rub`, `created_at`). Уникальность по
(`budget_id`, `period_start`, `threshold`) — предупреждаем **один раз на порог за период**,
а не на каждую задачу.

Проверка — в том же sweeper-цикле, что и прочие фоновые задачи воркера (интервал в
конфиг, дефолт 15 мин). Превышение → запись события + `log.warn` + отметка на дашборде.
**Обработка не останавливается** (решение владельца).

## 5. Отчёты

### 5.1 API

`GET /api/v1/billing/summary`

| Параметр | Значения |
|---|---|
| `from`, `to` | период (обяз.) |
| `group_by` | `organization` \| `project` \| `document_type` \| `provider` \| `day` |
| `cost_basis` | фильтр: `vendor` \| `amortized` \| `all` (дефолт `all`) |

Ответ: строки группировки с `total_rub`, `jobs_count`, `tokens_in`, `tokens_out`,
`ocr_pages`, `saved_rub` (кэш), `estimate_share` (доля задач с неизвестной ставкой —
показатель доверия к цифре).

`GET /api/v1/billing/budgets` — бюджеты с текущим расходом за период и статусом порогов.
`GET /api/v1/billing/job/:id` — разбор стоимости одной задачи (`cost_breakdown` как есть).

Доступ — только админ/владелец инстанса; данные по всем организациям видит владелец,
организация видит свои (tenant-scope как в остальных маршрутах).

### 5.2 UI

Экран «Расходы»: период, переключатель разреза, таблица + график по дням, плашка
бюджетов со светофором, отдельной строкой «сэкономлено кэшем». Стоимость одной задачи
раскрывается из существующей колонки в списке разборов.

## 6. Порядок работ

| Этап | Что | Почему в таком порядке |
|---|---|---|
| **1** | `provider_settings.rates` + `jobs.cost_breakdown`, `cost.ts` считает по провайдеру и пишет снимок | фундамент; без него отчёты считают по врущим глобальным ставкам |
| **2** | Заполнить ставки существующим провайдерам, `config.cost` → fallback для провайдеров без `rates` | обратная совместимость |
| **3** | `GET /billing/summary` + разрезы | закрывает основную боль — «почему столько» |
| **4** | Кэш-экономия (`would_cost`) | дешёвый и наглядный эффект |
| **5** | Стоимость в `llm_gateway_usage` | второй канал расхода |
| **6** | `budgets` + `budget_events` + sweeper | предупреждения |
| **7** | UI «Расходы» | поверх готового API |

## 7. Миграция существующих данных

Историю **не пересчитываем**: старые задачи не знают, каким провайдером обработаны.
`cost_breakdown` остаётся `NULL`, отчёт по таким периодам показывает `total_rub` из
`cost_rub` и помечает строку как «без разбора». Точка отсечения фиксируется датой
применения миграции — иначе получим смесь честных и додуманных цифр.

## 8. Приёмка

1. Задача, прошедшая через Yandex, имеет `cost_breakdown` с двумя провайдерами
   (Vision + AI Studio), сумма строк совпадает с `total` и с `cost_rub`.
2. Задача через провайдера без `rates` → `estimate = true`, а **не** нулевая стоимость.
3. Локальный провайдер попадает в отчёт с `cost_basis = amortized` и не смешивается
   с «деньгами наружу» при фильтре `cost_basis=vendor`.
4. `GET /billing/summary?group_by=organization` за июль воспроизводит цифры, которые
   в июле считались руками через SQL.
5. Повторная загрузка того же файла → `cache_hit: true`, `would_cost` заполнен,
   в отчёте растёт «сэкономлено».
6. Бюджет с `limit_rub` ниже текущего расхода → ровно одно событие на порог за период,
   обработка задач не останавливается.
7. Изменение ставки провайдера **не меняет** стоимость уже посчитанных задач.

## 9. Открытые вопросы

- **Валюта.** Anthropic считает в USD, остальное в ₽. На старте: храним валюту в строке
  расчёта, отчёт показывает итоги по валютам раздельно. Единый ₽-итог требует курса —
  нужен источник и политика фиксации (курс на дату задачи или на дату отчёта). Решать,
  когда появится заметный USD-расход.
- **Ставка для локального железа.** `amortized` подразумевает переводную цену
  (амортизация GPU + электричество / прогноз токенов). Числа нет — до его появления
  ставим `rates = NULL` и локальные задачи идут как `estimate`.
- **Гранулярность OCR.** Сейчас различаются «печатная страница» и «страница с таблицами».
  Если провайдер введёт другие тарифные позиции, `kind` в строке расчёта расширяется —
  формат это допускает без миграции.
