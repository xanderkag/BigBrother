# Приложение № ___ к Договору № ___ от «___» __________ 20__ г.

## Технические условия оказания услуг по распознаванию документов

---

### 1. Предмет и общее описание Сервиса

1.1. Исполнитель предоставляет Заказчику программный сервис автоматического
распознавания и структурирования документов (далее — «Сервис»). Сервис
принимает файл документа (скан, PDF или изображение), определяет тип
документа, извлекает значимые поля в структурированный вид (JSON) и
возвращает результат Заказчику с оценкой достоверности (confidence) по
каждому полю.

1.2. Оптическое распознавание и извлечение данных (OCR/LLM) относятся к зоне
ответственности Исполнителя. Сопоставление полученных данных с учётными
сущностями Заказчика и пользовательский интерфейс относятся к зоне
ответственности Заказчика.

---

### 2. Перечень поддерживаемых типов документов

2.1. **Базовый пакет (Этап 1):**

| № | Тип документа | Код типа (`document_type`) |
|---|---|---|
| 1 | Счёт на оплату | `invoice` |
| 2 | Счёт-фактура | `tax_invoice` |
| 3 | Универсальный передаточный документ (УПД) | `upd` |
| 4 | Товарно-транспортная накладная (форма 1-Т) | `ttn` |
| 5 | Транспортная накладная (форма 2013 г.) | `transport_invoice` |
| 6 | Международная транспортная накладная (CMR) | `cmr` |
| 7 | Акт выполненных работ / оказанных услуг | `services_act` |
| 8 | Путевой лист | `waybill` |
| 9 | Заявка на перевозку | `transport_request` |
| 10 | Платёжное поручение | `payment_order` |

2.2. **Пакет внешнеэкономической деятельности (Этап 2)** подключается
отдельным этапом по согласованию Сторон (внешнеторговый инвойс, упаковочный
лист, коносамент, таможенная декларация и иные документы ВЭД). Конкретный
перечень фиксируется дополнительным соглашением.

2.3. Сервис поддерживает создание **пользовательских типов документов** под
индивидуальную схему полей Заказчика.

2.4. Актуальный машиночитаемый перечень поддерживаемых типов всегда доступен
Заказчику по запросу `GET /api/v1/capabilities`.

---

### 3. Форматы и ограничения по входным файлам

| Параметр | Значение |
|---|---|
| Поддерживаемые форматы | PDF, JPG, PNG (базовые); XLSX, DOCX; TIFF, HEIC — по согласованию |
| Максимальный размер файла | до 50 МБ |
| Максимальное количество страниц | до 30 |
| Языки документов | русский, английский (иные — по согласованию) |
| Многодокументные файлы | поддерживается разбор PDF/XLSX, содержащих несколько документов; результат возвращается массивом `documents[]` |

---

### 4. Способ интеграции и состав API

4.1. Взаимодействие осуществляется по протоколу HTTPS, программный
интерфейс — REST API. Базовый адрес: **`https://<боевой-домен>/api/v1`**.

4.2. Основные методы API:

| Метод | Назначение |
|---|---|
| `POST /api/v1/jobs` | приём документа на обработку (загрузка файлом либо передача по защищённой ссылке `file_url`) |
| `GET  /api/v1/jobs/{id}` | опрос статуса и получение результата обработки |
| `POST /api/v1/jobs/{id}/reprocess` | повторная обработка ранее загруженного документа |
| `GET  /api/v1/capabilities` | перечень поддерживаемых типов документов и возможностей Сервиса |

4.3. **Доставка результата (webhook).** По завершении обработки Сервис
отправляет результат HTTP-запросом `POST` на адрес (`webhook_url`),
указанный Заказчиком. Обеспечивается повторная доставка при сбоях получателя
(до 5 попыток с нарастающей задержкой). Заказчик обеспечивает идемпотентную
обработку результата по идентификатору `job_id`. Дополнительно статус и
результат могут быть получены опросом `GET /api/v1/jobs/{id}`.

4.4. Структура полей результата и контракт webhook зафиксированы в
спецификации OpenAPI 3.1, передаваемой Заказчику как часть технической
документации.

4.5. **Версионирование контракта.** Добавление новых полей является обратно
совместимым изменением. Переименование или удаление полей выпускается как
новая версия контракта с параллельной работой обеих версий не менее
6 (шести) месяцев.

---

### 5. Аутентификация и безопасность обмена

5.1. **Входящие запросы Заказчика к Сервису.** Авторизация по персональному
токену доступа (Bearer-токен), выдаваемому Исполнителем Заказчику
индивидуально. Действуют ограничения по частоте запросов на токен. Токен
передаётся по защищённому каналу; плановый срок ротации токена —
6 (шесть) месяцев.

5.2. **Исходящие уведомления Сервиса (webhook) к Заказчику.** Каждый запрос
подписывается алгоритмом HMAC-SHA256 по телу запроса (заголовок
`X-Parsdocs-Signature`, общий секрет, выдаваемый индивидуально). Заказчик
обязан проверять подпись до обработки полученных данных.

5.3. Передача данных между Сторонами осуществляется исключительно по
защищённому каналу (TLS/HTTPS).

5.4. Секретные данные и ключи маскируются в журналах работы Сервиса и не
включаются в передаваемый Заказчику результат.

### 6. Размещение и обработка данных

| Параметр | Значение |
|---|---|
| Площадка размещения Сервиса | выделенная серверная инфраструктура Исполнителя по адресу `<контур/домен>` |
| Доступ к Сервису | по доменному имени, по защищённому каналу (TLS/HTTPS) |
| Срок хранения результатов и исходных файлов | 30 (тридцать) календарных дней по умолчанию, далее — автоматическое удаление; иной срок — по согласованию Сторон |
| Обезличивание данных | по запросу Заказчика (параметр `redact_pii`) персональные данные (ИНН, КПП, ФИО, телефон, e-mail, государственный регистрационный знак ТС, адрес) исключаются из доставляемого результата |
| Резервное копирование и восстановление | по регламенту Исполнителя |

### 7. Параметры качества и производительности

7.1. Стороны фиксируют целевые показатели качества, на которые Сервис
выходит поэтапно. На каждом этапе показатели подтверждаются по результатам
совместного измерения на согласованном наборе документов Заказчика.

7.2. **Этап 1 — запуск и опытная эксплуатация:**

| Параметр | Целевое значение |
|---|---|
| Время обработки одного документа | ≤ 90 сек |
| Точность по критичным полям (ИНН сторон, номер документа, сумма с НДС) | ≥ 90 % |
| Точность по прочим полям | ≥ 75 % |
| Пропускная способность | от 5 документов в минуту |
| Доступность Сервиса | 95 % |

7.3. **Этап 2 — промышленная эксплуатация:**

| Параметр | Целевое значение |
|---|---|
| Время обработки одного документа | ≤ 30 сек |
| Точность по критичным полям | ≥ 95 % |
| Точность по прочим полям | ≥ 80 % |
| Пропускная способность | до 60 документов в минуту |
| Доступность Сервиса | 99,5 % |

7.4. Сроки перехода между этапами и порядок измерения показателей
фиксируются в Плане ввода Сервиса в эксплуатацию (Приложение № ___).

7.5. При невозможности достоверно распознать поле Сервис возвращает значение
`null` (отсутствие данных), а не предполагаемое значение.

7.6. Статусы результата обработки:
- `done` — обработка завершена, результат пригоден для автоматической загрузки;
- `needs_review` — требуется проверка оператором (низкая достоверность либо
  расхождения при валидации);
- `failed` — ошибка обработки (причина передаётся в поле `error`).

---

### 8. Обработка персональных данных и соответствие законодательству

8.1. При обработке документов, содержащих персональные данные, Стороны
действуют в соответствии с Федеральным законом от 27.07.2006 № 152-ФЗ
«О персональных данных».

8.2. Обработка данных Заказчика производится на инфраструктуре,
расположенной на территории Российской Федерации. Трансграничная передача
данных без согласия Заказчика не осуществляется.

8.3. Исполнитель применяет организационные и технические меры защиты данных:
шифрование канала передачи (TLS), аутентификацию по токену, криптографическую
подпись уведомлений (HMAC-SHA256), маскирование секретов в журналах,
ограниченный срок хранения данных.

8.4. Распределение ролей Сторон в части обработки персональных данных
(оператор персональных данных / лицо, осуществляющее обработку по поручению)
определяется отдельным условием Договора либо соглашением об обработке
персональных данных.
