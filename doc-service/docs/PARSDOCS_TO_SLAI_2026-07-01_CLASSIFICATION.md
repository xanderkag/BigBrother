## Коротко: что нужно от вас

Одно контрактное изменение (аддитивное, back-compat), уже задеплоено:

1. Для неопознанных доков `document_type` приходит как **строка `"unknown"`**
   (не `null`, без отдельного флага) — явный маркер «тип не опознан», а не ошибка.
2. `schema_version` бампается **`1.0 → 1.1`** (MINOR = аддитивно, по политике Q17):
   `document_type` теперь может нести новое значение `"unknown"`.

Ваш код может **не меняться** и продолжит работать. Мы **просим** трактовать
`document_type: "unknown"` как «отложить на ручную/review», а не как fail.
Детали ниже.

## 1. Классификация стала точнее (хорошая новость + переотдача)

Мы переработали определение типа. Три сигнала теперь работают вместе:

- **имя файла как сигнал** — если имя буквально несёт тип (`Act_*`, `ТТН_*`,
  `VAT_invoice_*`, `988726MBL.xls`), это учитывается как weighted-сигнал/tie-breaker
  (не перебивает сильный заголовок в теле, но добивает слабый контент);
- **фикс regex-классификатора** — ключевое слово «АКТ» матчилось некорректно
  (двойное поедание пробела в «АКТ ОКАЗАННЫХ»), из-за чего акты уходили в
  `wire_transfer` / `customs_declaration`;
- **новый LLM-классификатор** (qwen3.6:27b) — прогоняется на **каждом** доке
  после OCR (~1s), выбирает один slug из нашего каталога (~38 типов), и **при
  любой ошибке/таймауте откатывается на keyword-классификатор**. Классификатор
  **никогда не роняет док**.

**Итог для вас:** `document_type` стал заметно надёжнее. Пофикшены типовые
мис-классификации: Акт → `customs_declaration`, ТТН → `invoice`,
УПД → `factInvoice` и т.п.

**Связь с переотдачей (Q16):** часть уже доставленных вам доков несла **неверный**
`document_type`. Исправленный тип придёт при **переотдаче** (та самая переотдача
восстановленных «пустых» доков по `job_id`, которую мы уже согласовали). То есть:
ждите (а) более надёжный `document_type` на новых доках, (б) исправленные типы на
переотданных.

## 2. НОВОЕ: `document_type: "unknown"` — единственное реальное изменение контракта

Раньше док, не подходящий ни под один тип каталога, всё равно получал какой-то
**уверенно-неверный** тип. Теперь такой док приходит честно как «не опознан»:

- **`document_type: "unknown"`** — строковый литерал, явный сигнал «не опознан».
  Мы больше не выдумываем тип и **не** используем `null` + отдельный флаг:
  по вашему согласованному выбору достаточно строки `"unknown"` (ваш receiver
  её нормализует и возвращает 201).

Это **осознанное состояние** («не опознан»), **не ошибка обработки** — сам док
извлечён и доставлен нормально, просто без уверенного типа. У нас в БД такой док
остаётся с `document_type = null` — `"unknown"` это только wire-представление.

### Влияние на контракт

- `schema_version` → **`1.0 → 1.1`**. Изменение **аддитивное, обратно совместимое**
  (§4.5 CONTRACT_TECH_APPENDIX): `document_type` теперь может нести новое значение
  `"unknown"`. По политике Q17 это **MINOR** — вам ничего не ломает.
- Envelope `version` остаётся `"v1"` (транспорт не меняется).
- Отдельного поля-флага **нет**: сигнал — само значение `document_type = "unknown"`.
  На опознанных доках `document_type` — реальный slug, как раньше.

### Что мы просим сделать (SHOULD, не MUST)

- **Трактовать `document_type: "unknown"`** как «отложить на ручную обработку /
  отправить в review», а не как отказ/fail.

Формально ваш код может остаться без изменений и не сломается. Но если сейчас
где-то есть жёсткое допущение «`document_type` всегда из каталога» — это как раз
тот случай, который стоит проверить.

## 3. Внутренняя наблюдаемость (FYI — НЕ контракт, действий не требует)

Мы завели пер-документную метаданную классификации для нашего оператора: метод
(`llm` / `keyword` / `filename` / `fallback`), время классификации (~1s),
кандидаты-претенденты, confidence. Это **чисто внутренний** job-API для нашего UI —
**в webhook-payload этого нет**, контракт этим не меняется, вы этого не видите и
ничего не делаете. Пишем одной строкой для контекста.

> _Опционально, если вам полезно:_ при желании мы могли бы позже отдавать в вебхуке
> сигнал уверенности классификации (hint «авто vs ручная») — если такой сигнал
> нужен вашему матчеру, скажите, добавим отдельно. Это не обязательство, а
> предложение.
